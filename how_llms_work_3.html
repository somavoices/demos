01
Phase One

Your message travels to the model

Before any AI processing begins, your text makes a physical journey across the internet to a specialised chip built for parallel computation.

A
You press Enter
HTTPS · Load balancer · GPU
+

Your browser packages your text into a secure HTTPS request — the same protocol used for online banking — and sends it to a data centre. A load balancer routes it to an available GPU: a chip originally built for video games, repurposed for AI because it runs millions of small calculations simultaneously.

YOU PRESS ENTER
      │
      ▼
[ Browser ]  — packages text as HTTPS request
      │  travels over the internet
      ▼
[ Data centre ]  — load balancer routes request
      │
      ▼
[ GPU assigned ]  — processing begins
🔌
Think of it this way: Your sentence is a document dropped in a secure courier envelope (HTTPS). It travels to a sorting office (data centre) and lands with the one specialist (GPU) who can process it — but only after every word is translated from English into the model's native language: pure numbers.
B
Text normalization
Unicode · Whitespace · Encoding
+

Before any AI work begins, your text is cleaned mechanically — extra spaces removed, invisible unicode characters stripped, encoding standardised. No intelligence yet.

RAW:   "What  is the  capital of France?​"
               ^^               ^^           ↑
          double space    double space    invisible char

CLEAN: "What is the capital of France?"

Why it matters: "café" typed two different ways (with a composed vs decomposed character) looks identical but would be treated as different inputs without this step.

C
Context window assembly
System prompt · History · Your message
+

The model never sees just your question. Before tokenization, everything is assembled into one document called the context window.

┌────────────────────────────────────────────────┐
│  CONTEXT WINDOW  (e.g. 200,000 tokens for Claude)  │
│                                                │
│  [System prompt]                              │
│  "You are Claude, a helpful assistant..."      │
│                                                │
│  [Conversation history]                       │
│  User: "Hi"   Assistant: "Hello!"              │
│                                                │
│  [Your current message]                       │
│  "What is the capital of France?"              │
└────────────────────────────────────────────────┘
         │
         ▼  All tokenized together as one input
📋
Think of it this way: The model has no memory between conversations. Every single time it responds, it must re-read the entire context window from scratch — like a chess player who can only see the current board position, not remember the previous game.
02
Phase Two

Text is converted into numbers

The model cannot read English. Everything must become numbers — tokens, then IDs, then meaning-encoded vectors ready for mathematical processing.

D
Tokenization
BPE · Subwords · Misspellings
+

Text is split into tokens — chunks the model treats as a single unit. A token is not always a whole word. Sometimes it's a word, sometimes part of a word, sometimes just punctuation, sometimes even a single character. The rules for splitting are decided before training (using algorithms like BPE or SentencePiece) and then frozen forever.

Every model tokenizes differently. The exact same sentence can split into different chunks and produce different IDs depending on which model you ask.

SAME SENTENCE — three different tokenizers:

  Input: "unbelievably refreshing"

  GPT-4 tokenizer:    ["un", "bel", "iev", "ably", " refreshing"]     5 tokens
  LLaMA tokenizer:    ["un", "believ", "ably", " refreshing"]          4 tokens
  Claude tokenizer:   ["unbeliev", "ably", " refresh", "ing"]          4 tokens

No single "right" answer. Each model learned its own splits
from its own training data.

The vocabulary is a fixed dictionary. When the model was being built, the tokenizer scanned huge amounts of text and decided which chunks were common enough to deserve their own slot in the dictionary. Common words — "the", "France", " is" — got their own slots. Rare or unusual sequences get split into smaller familiar pieces. There is no on-the-fly invention; every token must already exist in the vocabulary, otherwise it gets broken down further until every piece is recognized.

The same word always gets the same token ID. If "cat" appears five times in your sentence, you'll see token ID 8415 five times. The model doesn't store five separate copies — it stores one entry in the dictionary and points to it five times.

REPEATED WORDS reuse the same ID:

  Input: "The cat sat on the mat. The cat slept."

  Splitting:
    "The" " cat" " sat" " on" " the" " mat" "." " The" " cat" " slept" "."

  Token IDs:
    [791, 8415, 7731, 389, 279, 12384, 13, 578, 8415, 22011, 13]
       ↑     ↑                              ↑     ↑              ↑
       │     │                              │     │              │
       │     └── same ID 8415 reused for "cat"   │              │
       │                                          │              │
       └── notice: "The" (capital, sentence start) = 791
                   " The" (with leading space) = 578  ← different ID!
                                                        same word, but the
                                                        leading space changes it

  Period "." → 13 (used twice — same ID both times)

Two subtle things to notice above:

  • Leading spaces matter. "The" at the start of a sentence and " The" mid-sentence are different tokens with different IDs. The space is baked into the token itself — that's how word boundaries get preserved without extra metadata.
  • Punctuation is its own token. The period gets its own ID, separate from any word.

What happens with rare or unfamiliar words? The tokenizer breaks them into smaller pieces it does know — down to single characters if necessary. This is also how misspellings get handled.

SUBWORD SPLITS for rare and misspelled words:

  "refrigerator"      → ["refriger", "ator"]          2 tokens — common word, common pieces
  "hippopotamus"      → ["hippo", "pot", "amus"]      3 tokens — broken into known sub-pieces
  "antidisestablishmentarianism"
                       → ["anti", "dis", "establish", "ment", "arian", "ism"]   6 tokens

  "beautyful"  (misspelled) → ["beauty", "ful"]            model still recovers meaning
  "definately" (misspelled) → ["def", "in", "ately"]
  "recieve"    (misspelled) → ["rec", "ie", "ve"]

  "qwxyzpq"     (gibberish)  → ["q", "w", "x", "y", "z", "p", "q"]
                                  falls all the way down to single characters —
                                  the tokenizer never gives up; it always finds a way

The original example sentence — "What is the capital of France?" — uses only common words, so each word becomes exactly one token:

"What"tok 1
" is"tok 2
" the"tok 3
" capital"tok 4
" of"tok 5
" France"tok 6
"?"tok 7

Follow " France" (token 6) through the next stages — it becomes the ID 3723, then a vector encoding everything the model learned about France.

🔤
Think of it this way: The tokenizer is like a Lego instruction book. The model only knows a fixed set of Lego pieces — maybe 100,000 of them. Whatever you write, the tokenizer has to build it out of those exact pieces. Common things have a single big piece ready to go ("France"). Rare things have to be assembled from smaller pieces ("hippo" + "pot" + "amus"). Truly weird input gets built one tiny brick at a time. The tokenizer never gives up.
E
Token IDs
Vocabulary lookup · Numbers only from here
+

Every token is looked up in a fixed vocabulary dictionary of 50,000–128,000 entries and replaced with its unique number. From this point on, the model works only with numbers — never raw text.

VOCABULARY LOOKUP:

  "What"     →  9858
  " is"      →   374
  " the"     →   279
  " capital" →  6864
  " of"      →   315
  " France"  →  3723  ← follow this ID into Stage F
  "?"        →    30

SEQUENCE: [9858, 374, 279, 6864, 315, 3723, 30]

Vocab sizes: GPT-2: 50,257 · GPT-4: 100,256 · Claude: ~100K+ · LLaMA 3: 128,000
📚
Think of it this way: Every word in the model's vocabulary has a permanent seat number — "France" is always seat 3723. The model passes seat numbers around, never the words. Stage F is where each seat number gets its full meaning loaded — like opening the seat's envelope and finding a rich description of everything associated with that word.
F
Embeddings
Meaning vectors · GPS coordinates for words
+

Before we get into the math, here is the idea in plain terms. Think of every word as a point on a map — except this map has many directions, not just north-south and east-west. Each direction stands for some quality of meaning. Words that share qualities end up near each other.

Imagine a tiny version with just three directions: is-it-alive, is-it-sweet, and does-it-move. The simplest way to fill in the table is yes-or-no — 1 means yes, 0 means no.

STEP 1 — start with simple yes/no answers (1 or 0):

                        alive   sweet   moves
  "dog"            →  [    1   ,    0   ,    1     ]
  "cat"            →  [    1   ,    0   ,    1     ]
  "apple"          →  [    0   ,    1   ,    0     ]
  "banana"         →  [    0   ,    1   ,    0     ]
  "car"            →  [    0   ,    0   ,    1     ]
  "rock"           →  [    0   ,    0   ,    0     ]

Already useful: dog and cat have identical rows, so the model
sees them as "the same kind of thing." Apple and banana match
each other. Rock matches nothing. This is the basic idea.

But yes/no is too crude. A dog isn't just alive — it's very alive in a way a tree isn't. A banana is sweeter than an apple. To capture how much of each quality, we let the numbers be fractions between 0 and 1.

STEP 2 — let numbers be fractions to capture "how much":

                        alive   sweet   moves
  "dog"            →  [  0.95 ,  0.05 ,  0.90  ]   ← strongly alive, barely sweet, moves a lot
  "cat"            →  [  0.95 ,  0.05 ,  0.85  ]   ← almost identical to dog, slightly less mobile
  "tree"           →  [  0.60 ,  0.10 ,  0.00  ]   ← alive, but not the same way a dog is
  "apple"          →  [  0.40 ,  0.75 ,  0.00  ]   ← partly alive (it's a fruit), pretty sweet
  "banana"         →  [  0.40 ,  0.90 ,  0.00  ]   ← like apple, but sweeter
  "car"            →  [  0.00 ,  0.00 ,  0.95  ]   ← not alive, not sweet, definitely moves
  "rock"           →  [  0.00 ,  0.00 ,  0.00  ]   ← none of these apply

Now the model can tell that:
  • dog and cat are nearly identical (rows differ by 0.05)
  • apple and banana are close, but banana is sweeter
  • tree is "alive" but a different kind of alive than dog
  • car shares "moves" with dog but nothing else

Where do these fraction values come from? We didn't pick them. The model finds them on its own during training. It starts with random numbers in every cell, then reads billions of sentences. Every time it sees "the dog ran" and "the cat ran," it nudges dog and cat closer together. Every time it sees "I ate an apple" and "I ate a banana," it nudges apple and banana closer. After billions of nudges, words that appear in similar contexts end up with similar numbers. Nobody ever told the model what "alive" or "sweet" means — those labels are our guess at what an axis ended up representing.

A REAL MODEL doesn't use 3 axes — it uses thousands.
Most of them have no human-namable meaning at all.
Some axes the model invents:

  Axis 47:    "is this a question word?"           → who, what, why score high
  Axis 142:   "is this formal or casual?"          → "thou" vs "u" land far apart
  Axis 891:   "is this about food?"                → pizza, sushi, banana score high
  Axis 2,034: ???                                     → high for: river, Tuesday, blue
                                                  no human can say what this captures

The model doesn't need us to label them. It just needs the
numbers to make good predictions.
🗺
Think of it this way: If you only had two axes — alive on the y-axis and sweet on the x-axis — you could literally plot these words on graph paper. "dog" and "cat" would land in the upper-left corner, almost touching. "apple" and "banana" would sit in the lower-right, also touching. "rock" would be at (0,0). That closeness is what "similar meaning" looks like to the model — it's just distance on a map.

That's the whole idea. What follows is the same thing in the language a real model speaks — token IDs looked up in a giant table called the embedding matrix, with thousands of numbers per row instead of three.

STAGE E output  →  STAGE F: embedding matrix lookup

  ID 9858 ("What")    → [ 0.21, -0.40,  0.88, -0.12,  0.55 ]
  ID  374 (" is")     → [ 0.05,  0.11, -0.22,  0.70, -0.08 ]
  ID 3723 (" France") → [ 0.78,  0.62,  0.15,  0.33, -0.50 ]
                            ↑ this vector carries "France-ness"
  ID 3723 in a different sentence about food?
    → same lookup, same starting vector
    → but attention (Stage I) will reshape it based on context

WHY SIMILAR WORDS GET SIMILAR VECTORS:
  "Paris" ID → [ 0.74,  0.68,  0.18,  0.30, -0.48 ] ← close to France
  "banana" ID → [-0.12, -0.55,  0.71, -0.23,  0.84 ] ← far away

SIZES: GPT-2: 768 · GPT-3: 12,288 · LLaMA 3 (8B): 4,096 · Claude: undisclosed
🗺
Think of it this way: The ID is just an address — it tells the model which row to pick from the embedding table. The vector at that row is what the model actually works with: a fingerprint of meaning, built by learning from billions of sentences. "France" and "Paris" have similar fingerprints. "France" and "banana" do not. Attention in Stage I then updates that fingerprint based on what surrounds it.

⚠ The problem with embeddings: they absorb human bias

Embeddings are learned from real human text — which means they inherit every pattern, stereotype, and bias present in that text. This was first discovered in Word2Vec (an early embedding system from 2013) and remains a live issue in modern LLMs.

WORD2VEC ARITHMETIC — what researchers found:

  "king" − "man" + "woman"  =  "queen"       ← gender relationship encoded
  "Paris" − "France" + "Italy"  =  "Rome"   ← capital relationship encoded

SO FAR SO GOOD. BUT ALSO:

  "programmer" − "man" + "woman"  =  "homemaker"
  "doctor" − "man" + "woman"      =  "nurse"
  "brilliant"  sits close to "man", far from "woman"

WHY THIS HAPPENS:
  Training text reflects historical writing patterns.
  If most articles about programmers used "he," the model
  learns that "programmer" and "man" belong together.
  The model did not invent this bias — it faithfully absorbed it.

This matters because these starting vectors feed into everything the model does. Biased embeddings produce biased translations, biased autocomplete, and biased answers. Modern LLMs use techniques to reduce this — but no model is fully free of it, because no training dataset is.

What you see above is from Word2Vec (2013). Modern LLMs use contextual embeddings — the same word gets different vectors depending on the sentence around it — which reduces but does not eliminate bias inherited from training data.
G
How similarity is measured
Background concept · Cosine · Euclidean
+
This is a background concept — not a step the model takes with your text. Cosine similarity is used in retrieval systems built around LLMs. Inside attention (Stage I), a related but different measure — the raw dot product — is used.
COSINE SIMILARITY  — angle between vectors (retrieval, search)

  "King"   → [ 0.9,  0.3 ]
  "Queen"  → [ 0.8,  0.4 ]   ← similar direction → score: ~0.99
  "Banana" → [-0.2, -0.9 ]   ← opposite direction → score: ~-0.1
  Range: -1 (opposite) to +1 (identical direction)

EUCLIDEAN DISTANCE  — straight-line gap (clustering, retrieval)

  "Paris" → "France" = 1.2  ← close
  "Paris" → "banana" = 8.7  ← far

Inside the model: attention uses raw dot product (Q·K / √d_k)
— related to cosine similarity but NOT identical. See Stage I.
H
Positional encoding
Word order · RoPE in modern models
+

Embeddings carry meaning but no sense of order. "Dog bites man" and "Man bites dog" have identical tokens in different positions — yet mean opposite things. Positional encoding adds a signal to each embedding marking its slot number.

WITHOUT positional encoding:
  ["France", "capital", "is"]  →  order unknown — could mean anything

WITH positional encoding:
  ["France" + pos_1]  [" is" + pos_2]  ["capital" + pos_3]
  "Dog bites man"  ≠  "Man bites dog"  ← now distinguishable

Original transformer (2017): fixed sinusoidal encoding
Modern models (LLaMA, likely Claude): RoPE — learned, relative
Both achieve the same goal: teaching the model about word order.
03
Phase Three

Every word reads every other word

Before transformers, AI models read one word at a time left-to-right and forgot early words by the end of a sentence. The transformer solved this by letting every word look at every other word at once — so "France" can pick up context from "capital of" and become "the political France" rather than "the cuisine France." This block repeats anywhere from 12 times in smaller models to over 100 times in the largest — each pass making the understanding deeper.

I
Attention — words borrowing meaning from each other
How the model figures out what relates to what
+

Going into this stage, every word still has the generic meaning it picked up from the embedding table in Stage F. " France" carries everything the model has ever learned about France — geography, food, language, history, capital, all mixed together. " capital" is similarly broad — it could mean a city, a financial capital, or even a capital letter. These vectors have not yet been shaped by this specific sentence.

Attention's only job here is to let each word borrow context from the others. It does not pick an answer. It does not produce "Paris." It just sharpens each word's meaning based on its neighbors, so the rest of the network has a context-aware understanding to work with. The actual answer-prediction happens many stages later.

WHAT EACH WORD IS BEFORE AND AFTER ATTENTION:

  " France"
    Before:  generic France — country, food, language, history, capital, art...
    After:   ↑ "capital" and "of" pull the meaning toward
             government / political-geography aspects of France.
             The food, art, language meanings get dimmed down.

  " capital"
    Before:  ambiguous — city-capital? financial-capital? capital-letter?
    After:   ↑ "of France" pulls it firmly toward "city-capital."
             The financial and grammar meanings get dimmed down.

  "What"
    Before:  generic question word.
    After:   ↑ links to "capital" — marks "capital" as the thing being asked about.

  "?"
    Before:  generic punctuation.
    After:   ↑ confirms this whole thing is a question — the model should ANSWER,
             not continue the sentence.

So when this stage finishes, the word " France" inside the model is no longer "generic France" — it's "France considered as a country whose capital is being asked about." The word " capital" is no longer ambiguous — it's "the city-capital, specifically of France." The model has not chosen Paris yet. It has just made the sentence's meaning unambiguous so the later stages can do their work.

How the borrowing actually works — the math, in plain English

Each word holds its embedding vector from Stage F — its meaning fingerprint. To decide how much one word should borrow from another, the model takes those two fingerprints and measures how well they point in the same direction. Two words whose vectors point similarly get a high score. Two words pointing in different directions get a near-zero score.

SCORING " France" against " capital" — step by step:

  " France"  has a vector: [ 0.78,  0.62,  0.15,  0.33, -0.50 ]
  " capital" has a vector: [ 0.71,  0.55,  0.20,  0.40, -0.35 ]

  Multiply each pair, then sum:
  (0.78×0.71)+(0.62×0.55)+(0.15×0.20)+(0.33×0.40)+(-0.50×-0.35)
  =  0.55  +  0.34  +  0.03  +  0.13  +  0.18
  =  1.23   ← high score: these vectors point in similar directions
                  the model learned this from seeing "capital of France"
                  countless times during training

  Compare to " France" against "What":
  "What"     has a vector: [ 0.21, -0.40,  0.88, -0.12,  0.55 ]
  (0.78×0.21)+(0.62×-0.40)+(0.15×0.88)+(0.33×-0.12)+(-0.50×0.55)
  =  0.16  +  -0.25  +  0.13  +  -0.04  +  -0.28
  =  -0.28  ← low / negative score: vectors don't align —
                  "What" doesn't add useful context to "France."

  Repeat this scoring for every pair of words in the sentence.
  Convert all scores to percentages so each word's borrowing-budget sums to 100%.

That's the entire mechanism. No grammar rules, no parser, no dictionary lookups. Just measuring how well meaning-fingerprints align, then borrowing in proportion. The model learned which directions to align by reading billions of sentences — every time it saw "capital of [country]" patterns, it nudged "capital" and country-vectors to point more similarly.

Think of it this way: Imagine each word as a person at a party, holding a name tag. The name tag starts off generic — "France" just says "France." During attention, each person looks around the room, decides who matters most for understanding what they should be right now, and quietly rewrites their own name tag to reflect that. After this step, "France" is no longer wearing a generic tag. It's wearing a tag that reads "the country whose capital is being asked about." Same word. Different, sentence-specific meaning. That richer name tag is what travels into the next stage — but no answer ("Paris") has been chosen yet.

When it gets harder — multiple references, long distance, ambiguity

The France example is clean — short sentence, no ambiguity. Real text is messier. Consider this passage:

"The lawyer told her client that she would call her back after she
reviewed the case, because she wanted to make sure she had all the facts."

  There are FIVE "she" pronouns and TWO women (lawyer, client).
  Every "she" needs a separate decision — and the right answer
  changes from one "she" to the next:

  "she would call her back"        →  lawyer     (the one doing the calling)
  "her back"                       →  client     (the one being called)
  "after she reviewed the case"     →  lawyer     (lawyers review cases)
  "because she wanted to make sure"  →  lawyer     (matches the reviewer)
  "she had all the facts"           →  lawyer     (same chain of logic)

Notice what attention has to do here. It can't just grab the nearest noun — "client" is closer to most of the "she" pronouns than "lawyer" is. It can't follow a single rule because the answer changes from one pronoun to the next. What it actually does is score every "she" against every earlier word, and the patterns it learned during training tell it that lawyers (not clients) review cases, that the subject of "told" is usually the one who follows up, and that "she" inside a "because" clause usually refers back to the actor of the main clause.

Three things make this hard, and attention handles all three at once:

  • Multiple candidates. Two women in the sentence, both grammatically valid for every "she."
  • Long distance. The fifth "she" is 20+ words away from "lawyer," but still gets linked to it.
  • The right answer shifts. The first "her" means client; the next four "she"s mean lawyer. Same word, same sentence, different referents.

And this is happening for every word in the sentence simultaneously — not just the pronouns. "told," "reviewed," "wanted," "facts" — each one is also asking who and what it relates to. Real models run this scoring across many parallel "lenses" at once: one lens might track grammatical roles (who is the subject?), another tracks topic (this is about a legal case), another tracks tone (this is formal). All these readings get blended together to form each word's final, context-aware meaning.

This is also why models occasionally get pronouns wrong in genuinely ambiguous sentences — the score for the wrong noun was just barely higher than the score for the right one. Humans use real-world knowledge to disambiguate; the model only has the patterns from training. When training patterns don't clearly favor one answer, attention picks whichever scored higher, and that's not always the human-correct choice.
J
The knowledge layer — recalling what the model learned
Where training-time facts get applied to your sentence
+

First — what is a "layer"? Think of it as one full pass of processing. Each transformer block is a layer, and one layer = one round of attention (Stage I) plus one round of knowledge retrieval (this stage). Models stack anywhere from 12 to over 120 of these blocks. Each layer takes the vectors from the previous layer, refines them slightly, and passes them on. A vector goes in, a vector comes out. No words are chosen along the way.

HOW MANY LAYERS DO REAL MODELS HAVE?

  Model                    Layers   What it means
  ─────────────────────────────────────────────────────
  GPT-1 (2018)               12    Early research model
  GPT-2 small (2019)         12    First public GPT
  BERT base (2018)           12    Google's reading model
  BERT large (2018)          24    Larger reading model
  LLaMA 3 8B (2024)          32    Runs on a laptop GPU
  LLaMA 3 70B (2024)         80    High-end server model
  GPT-3 175B (2020)          96    Original ChatGPT base
  LLaMA 3 405B (2024)       126    One of the largest open models
  ─────────────────────────────────────────────────────
  Claude: number of layers not publicly disclosed by Anthropic.

  Each layer = one more pass of attention + knowledge retrieval.
  More layers = richer understanding, but slower and costlier to run.

So what does the knowledge half of each layer actually do?

After attention has decided which words relate to which, the knowledge layer does the next half of the work: it pulls in associations the model learned during training. If attention has marked the vector at the "capital" position as "capital, of, France-context," the knowledge layer pumps related concepts into that vector — French geography, European cities, political-capital associations, and yes, Paris-related associations.

But — and this is the part to internalize — this layer does not select a word. It does not commit to "Paris." It enriches the vector. The vector becomes a denser, more informed list of numbers. Whether the final answer turns out to be Paris, or "I don't know," or something else entirely is determined many stages from now, after the full stack of layers finishes and several more steps run on top of the result.

WHAT ONE LAYER ACTUALLY DOES TO THE "capital" VECTOR:

  Vector at "capital" position, going IN to layer:
    [ 0.71,  0.55,  0.20,  0.40, -0.35, ... ]
    Meaning: "the city-capital, in the context of France"
    (this much was already established by attention)

  Layer applies thousands of learned weights:
    Pattern "capital + France" is recognized — strongly active.
    Pattern "capital + Germany"             — barely active.
    Pattern "capital + UK"                  — barely active.
    Many other patterns — most barely active.

  Vector at "capital" position, coming OUT of layer:
    [ 0.74,  0.61,  0.29,  0.45, -0.30, ... ]
    Same shape, different numbers. The vector now leans MORE
    toward Paris-adjacent regions of meaning-space, but it is still
    a vector — not a word. No selection has been made.

  ────────────────────────────────────────────
  This is what every layer does, again and again.
  Layer 1 nudges. Layer 2 nudges further. Layer 50 nudges further still.
  After the FINAL layer, the vector is at its most informed.
  Even then, no word has been chosen — that comes later (Stage M onward).

Why does Paris-association get strong and Berlin-association stay weak?

During training, the model read billions of sentences containing "capital of France" near "Paris," and almost none saying "capital of France... Berlin." Over time, the weights in this layer adjusted so the combination "capital + France" produces a strong nudge in the Paris direction — and a weak one in the Berlin direction. The model was not programmed with this rule. It learned the pattern from repetition. Think of it like muscle memory built from practice — the path "capital of France → Paris" became well-worn from being seen so often, while "capital of France → Berlin" stayed faint because it was never reinforced.

This is also why models can be wrong on rare facts. If "capital of [some small country] → [its capital]" barely appeared in training, those weights are weak, and the layer cannot push the vector strongly in any particular direction. When that happens, the later sampling stage may end up picking something plausible-sounding but wrong — what people call a hallucination.
🧠
Think of it this way: Attention told the model what relates to what in this sentence. The knowledge layer tells the model what it knows about those relationships — and pours that knowledge into each vector. After both halves of one layer, each word's vector is slightly richer. After 50 layers, much richer. But all the way through, it stays a vector. The vector is the model "thinking." Word selection is a separate, much later step.
K
Keeping the signal alive across many layers
Why 100+ layers don't destroy what earlier layers learned
+

This stage is supporting machinery — not a step that decides anything. Its job is to keep the math stable and the original meaning intact while 100+ layers stack on top of each other. The vector that comes out of this stage is still the same vector — just protected from being destroyed.

The transformer runs anywhere from 12 to over 120 layers deep depending on the model. That raises a real problem: if each layer rewrites the vector completely, the original meaning of "France" could be unrecognisable by layer 50. Two mechanisms prevent this.

Residual connection — the "keep a copy" rule

Before each layer does its work, the model takes a copy of the current vector and sets it aside. After the layer finishes, that copy is added back to the result. So no layer can ever fully erase what came before — it can only add a correction on top.

EXAMPLE: the "France" vector passing through one layer

  Vector coming in (after layer 3):
    [ 0.78,  0.62,  0.15,  0.33, -0.50 ]
    "France" — country, western Europe, Paris nearby

  What layer 4's attention adds:
    [ 0.02, -0.05,  0.11,  0.08,  0.03 ]
    (small nudge — "this sentence is a question")

  WITHOUT residual connection:
    Layer 4 overwrites:  [ 0.02, -0.05,  0.11,  0.08,  0.03 ]
    "France-ness" is gone. Layer 5 sees nothing useful.

  WITH residual connection  (input + layer output):
    [ 0.78,  0.62,  0.15,  0.33, -0.50 ]   ← original copy
  + [ 0.02, -0.05,  0.11,  0.08,  0.03 ]   ← layer's correction
  = [ 0.80,  0.57,  0.26,  0.41, -0.47 ]   ← refined, France still present

  Layer 5 receives a richer vector. "France" is still in there.

Layer normalisation — the "steady volume" rule

After many layers of additions, some numbers in the vector can grow very large and some shrink very small. A vector like [412, -0.04, 8831, 1.1] is unstable — the huge value 8831 would dominate all calculations. Layer normalisation rescales every vector after each layer so all values sit in a similar small range, keeping the maths stable throughout the whole stack.

LAYER NORMALISATION — rescaling to a consistent range

  Unstable vector (after many additions):
    [ 412.3,  -0.04,  8831.2,  1.1 ]
    ← 8831 would dominate every calculation that follows

  After layer normalisation (mean=0, range compressed):
    [   0.2,   -0.8,     1.7,  0.0 ]
    ← all values in a similar range — stable input for next layer

  This runs before every sub-layer, every pass, across all layers.
  Without it, the model's maths would become numerically unstable
  by around layer 10 and training would fail entirely.
🔄
Think of it this way: Imagine passing a document through 96 editors. Without the copy rule (residual), each editor rewrites from scratch and the original meaning is gone by editor 10. With the copy rule, each editor only writes their additions in the margin — the original is always there underneath. Layer normalisation is like making sure no editor uses a font so large it covers everything else.
L
Remembering what was already calculated
Why long responses don't slow down token by token
+

This stage is supporting machinery — not a step that decides anything. It is a memory optimization that becomes important once the model starts generating response words (covered in Stage Q). Skip ahead and come back if you want; the answer is not chosen here.

The model generates one word at a time. Each new word needs to run attention — comparing itself against every previous word. Without any shortcut, generating word 50 would require recalculating everything for words 1 through 49 all over again. And word 100 would redo everything again. That would be extremely slow.

The solution: save the attention calculations for every word as soon as they are done. When the next word is generated, those saved results are reused — only the newest word needs fresh calculation. This saved store is called the KV cache (K and V are the two sets of numbers each word computes during attention — think of them as "what I offer" and "what I contain").

GENERATING: "The capital of France is Paris."

  Word 1 "The":      calculate, save.  Attend to: [The]
  Word 2 "capital":  calculate, save.  Reuse saved "The".
  Word 3 "of":        calculate, save.  Reuse saved "The", "capital".
  Word 4 "France":    calculate, save.  Reuse all 3 saved.
  Word 5 "is":         calculate, save.  Reuse all 4 saved.
  Word 6 "Paris":      calculate, save.  Reuse all 5 saved.

  Without this:  word 100 recalculates words 1–99 from scratch. Very slow.
  With this:     word 100 only does its own calculation. Fast.

  TRADE-OFF: the saved store takes up GPU memory.
  Very long conversations use more memory — there is a limit.
📝
Think of it this way: Imagine writing an essay and needing to re-read the entire essay from page 1 before writing each new sentence. The KV cache is like keeping sticky notes — every sentence you finish, you write a quick summary note. The next sentence only reads the new notes, not the whole essay again.
04
Phase Four

Turning understanding into a word

After all the transformer layers, each word's vector is rich with meaning. But the model still hasn't produced any output text. Phase 4 is the translation back — from a meaning-vector to an actual word choice. It happens one word at a time, every single step of the response.

M
From the final vector to a score for every word
Where the model finally turns understanding into a list of word scores
+

After all the transformer layers have run — every layer of attention, every layer of knowledge retrieval — the model holds one vector for the last position in the sentence. This vector is the result of the entire pipeline so far. It is not a raw embedding anymore; it has been refined by every layer. It now encodes a compressed summary of: the question being asked, the France context, the kind of answer expected.

Here is the moment the model commits to anything. Up until now, no word has been chosen. The vector has been getting richer and more informed, but it has stayed a vector. This stage is where the model finally translates its internal "thinking" into a list of word scores — one score per word in its 100,000-word vocabulary. These scores are called logits.

WHAT THE FINAL VECTOR ENCODES — built up across all layers:

  Input was: "What is the capital of France?"

  Layer 1:    detected this is a question (from "What" + "?")
  Layer 4:    identified "capital" as the key noun being asked about
  Layer 8:    linked "capital" to "city" associations
  Layer 15:   connected "France" to European-geography associations
  Layer 24:   pulled in "capital of France" associations strongly
  Layer 32+:  refined confidence, dimmed associations with wrong countries

  FINAL VECTOR at the "?" position:  [..., ..., ...]  (thousands of numbers)
  Encodes: "the answer is a major European city, specifically the capital
            of France, high confidence, proper noun expected."

  This vector has never said "Paris" yet.
  It is still numbers — just very meaning-loaded numbers.

How the vector becomes word scores. The model asks: "of all the 100,000 words in my vocabulary, which one does this meaning-vector point most toward?" Every word in the vocabulary also has a vector (the same embedding table from Stage F, used in reverse). The model measures how well the current meaning-vector aligns with each vocabulary word's vector, and produces one raw score per word.

SCORING EVERY VOCABULARY WORD against the final vector:

  "Paris"   →  14.8   ← vector strongly aligns — clear top candidate
  "London"  →   9.2   ← also a capital, but wrong country
  "Berlin"  →   8.1   ← also a capital, but wrong country
  "Rome"    →   7.4   ← also a capital, but wrong country
  "the"     →   3.1   ← not a city, low score
  "banana"  →  -5.3   ← completely unrelated, negative score

  ... scored against all ~100,000 vocabulary words this way

  These are still raw scores. They cannot be used directly to pick a word
  because they are on no particular scale. The next stage fixes that.
🔍
Think of it this way: Up until now the model has been refining a thought in its own internal language. This is the stage where it finally lays that thought next to every word in its dictionary and asks "which of you matches what I'm thinking?" Paris matches strongly. Banana doesn't match at all. The output is one number per word — a similarity score for the entire vocabulary. No word is picked yet, but for the first time the candidates are sorted.
N
Converting scores into percentages
Why raw scores aren't enough · Making a fair comparison
+

Stage M produced raw scores — Paris got 14.8, London got 9.2. But raw scores have a problem: you cannot compare them directly or use them to make a choice. Is 14.8 a lot better than 9.2, or just a little? What does 14.8 even mean? There is no scale.

The solution is to convert all scores into percentages that sum to 100%. Now the comparison is clear: Paris has a 72% chance of being the right next word. That is meaningful. This conversion is called softmax.

The important thing softmax does: it amplifies the differences. A score of 14.8 versus 9.2 is not just "a bit higher" — after softmax it becomes six times more likely. The model becomes confident rather than wishy-washy.

Notice: all five percentages add up to 100%. Now the model can make a decision. Stage O decides how to pick from these percentages.

🎯
Think of it this way: Imagine a panel of judges each holding up a raw score — 14.8, 9.2, 8.1. Those numbers alone don't tell you what to do. But if you convert them to "Paris wins 72% of the vote," you have something you can act on. Softmax is the vote-counting step that turns raw scores into a usable percentage breakdown.
O
How the model picks the actual word
Always pick the top? Or leave room for variety?
+

Stage N produced percentages — Paris 72%, London 12%, Berlin 9%. Now the model must pick one word. The obvious choice is always pick the highest. But that produces robotic, repetitive responses. So there are several strategies, each making a different trade-off between accuracy and variety.

ALWAYS PICK THE WINNER (Greedy)
  Always choose the word with the highest percentage.
  → Always "Paris". Every time. No variation.
  Good for: factual questions, code, structured output.
  Problem: creative writing becomes repetitive and flat.

──────────────────────────────────────────────────────

PICK FROM THE TOP FEW (Top-K)
  Pick randomly from only the top K candidates.
  K = 3: randomly choose from Paris(72%), London(12%), Berlin(9%).
  Could pick London — adds human-feeling variation.
  Problem: on uncertain questions K might include nonsense words.

──────────────────────────────────────────────────────

PICK FROM THE MEANINGFUL CANDIDATES (Top-P)
  Keep adding words until the percentages reach 90% total,
  then pick randomly from just those words.
  Confident answer: Paris(72%) alone hits 72% — very narrow pool.
  Uncertain answer: spreads across many words — wider pool.
  This automatically adjusts to how confident the model is.
  Claude uses this — it is the most natural-feeling approach.

──────────────────────────────────────────────────────

THE CREATIVITY DIAL (Temperature)
  Before any of the above, squash or spread the percentages.

  Low temperature (cautious):   Paris 98%,  London 1.5%  → very safe
  High temperature (creative):  Paris 38%,  London 28%   → more varied

  Claude uses low temperature for code (correctness matters).
  Higher temperature for creative writing (variety matters).
🎲
Think of it this way: Imagine a lottery where Paris has 72 tickets, London has 12, Berlin has 9. Greedy means the highest ticket-holder always wins — predictable but boring. Top-P is like drawing from only the tickets that make sense together. Temperature is like adding or removing tickets before the draw — fewer tickets for Paris means other cities get a fairer shot, making the response feel more creative and varied.
P
Knowing when to stop
How the model decides the answer is complete
+

The model does not have a built-in sense of "I have finished answering." Left alone, it would keep generating words forever — because there is always a next most-likely word. Something has to tell it to stop.

During training, every piece of text the model learned from had an end marker — a special token meaning "this is where this document ends." The model saw this millions of times. It learned that after a complete answer, the next most likely token is that end marker. So when generating, it eventually picks the end marker as the highest-scoring token — and stops.

THE RESPONSE GENERATING:

  "The"  → "capital"  → "of"  → "France"  → "is"  → "Paris"  → "."
       ↓
  Next token scores:
    [END]     →  9.2   ← highest — sentence feels complete
    "It"      →  4.1   ← could keep going
    "The"     →  3.8   ← could start a new sentence

  Model picks [END] → generation stops.

THREE WAYS GENERATION ENDS:
  1. Model picks the end token        ← learned, most natural
  2. A maximum word limit is reached  ← safety cap set by the system
  3. A custom signal is matched       ← e.g. "stop when you see User:"
🔚
Think of it this way: The model learned to stop the same way it learned everything else — from examples. Every book, article, and conversation it trained on had an ending. It absorbed the pattern that a complete, well-formed answer is followed by silence. Generating the end token is the model's way of saying "I'm done here."

05
Phase Five

The response travels back to you

One token at a time, streamed live to your browser, then converted back into readable English.

Q
The loop — how each word of the response actually gets made
The whole pipeline runs once per output word
+

Here is the part that often gets glossed over. Everything from Stage F through Stage P runs one full time per word the model outputs. The model does not produce a whole sentence in a single pass. It produces one word, appends that word to its input, runs the entire pipeline again, produces the next word, appends that, runs again — until it picks the stop token.

So for the answer "The capital of France is Paris." (8 tokens including the period and stop), the pipeline runs eight separate times. Each run:

  • Sees the original question plus every output word produced so far.
  • Runs all the layers of attention and knowledge retrieval over the whole sequence.
  • Produces logits for the entire vocabulary, applies softmax, and samples one word.
  • Appends that word to the sequence and starts again.

The KV cache from Stage L is what makes this practical. Without it, every run would redo all the previous runs from scratch.

FULL TRACE — generating "The capital of France is Paris." token by token.
Each row is one complete run of the entire pipeline (Stages F–P).

  Run 1.  Input to model:
            "What is the capital of France?"
          Pipeline runs over 7 input tokens.
          Final-position vector → logits → softmax → sample.
          Top candidates: "The" 41% · "Paris" 28% · "It" 9% · "France" 5% · ...
          Picks: "The"
          New sequence: "What is the capital of France? The"

  Run 2.  Input to model:
            "What is the capital of France? The"
          Pipeline runs again. KV cache reuses runs 1's work for the
          first 7 tokens — only the new "The" position computes fresh.
          Top candidates: "capital" 67% · "answer" 12% · "city" 7% · ...
          Picks: " capital"
          New sequence: "What is the capital of France? The capital"

  Run 3.  Top candidates: "of" 84% · "is" 8% · ...
          Picks: " of"

  Run 4.  Top candidates: "France" 91% · "this" 3% · ...
          Picks: " France"

  Run 5.  Top candidates: "is" 78% · "would" 6% · ...
          Picks: " is"

  Run 6.  Input now: "What is the capital of France? The capital of France is"
          This is the moment "Paris" finally gets selected — at run 6.
          Up until this run, "Paris" was never the chosen word.
          The earlier runs were building the SCAFFOLDING of the sentence.
          Top candidates: "Paris" 88% · "the" 4% · "located" 3% · ...
          Picks: " Paris"

  Run 7.  Top candidates: "." 79% · "," 14% · " and" 4% · ...
          Picks: "."

  Run 8.  Top candidates: [STOP] 92% · " It" 3% · " The" 2% · ...
          Picks: [STOP] → loop ends.

FINAL OUTPUT: "The capital of France is Paris."

Notice when "Paris" actually appears. It is not at run 1. It is not at the end of the transformer layers in any earlier stage. It is at run 6 — after the model has already committed to "The," "capital," "of," "France," and "is." Each of those earlier choices shaped what came next. If run 1 had sampled "Paris" instead of "The," the sentence would have started differently — perhaps "Paris is the capital of France." The model committed to a sentence shape first, and the actual fact-word came later.

What state survives between runs? Two things, and only two things:

  • The token sequence so far. The original question plus every output token already chosen. This is the model's complete "memory" of the conversation. Nothing else carries over — not the vectors, not the layer activations, not the logits.
  • The KV cache. The attention computations for tokens that have already been processed. This is purely an optimization — the answer would be identical without it, just slower.

Everything else — the embeddings at Stage F, the attention scores at Stage I, the layer-by-layer refinements through Stage L, the final vector at Stage M, the logits, the softmax percentages — all of that is recomputed from scratch every single run. The model does not "remember" that it was confident about Paris on run 6 when it gets to run 7. It just sees the new sequence and starts over.

This is why the model cannot revise. Once a token is sampled and appended to the sequence, it is part of the input on the next run. There is no backspace, no edit, no second-guessing. If the model picks a wrong word at run 3, runs 4 onward have to live with that choice — they will work to make the sentence sensible given the wrong word, not retry. This is also why prompt wording matters from the very first token: the first run is committed to before any later run has a chance to influence it.
⌨️
Think of it this way: Imagine a typist with no backspace key, no clipboard, and a strange habit. Before typing each new letter, they re-read the entire document from the beginning, think hard about what should come next, and only then type one more letter. Then they re-read everything again — including the letter they just typed — before typing the next one. Slow. But this re-reading is exactly how each word the model writes is informed by every word that came before, including the words it just produced moments ago.

Frequently asked questions

What is a prompt?
+
A prompt is everything inside the context window — your question, the system instructions, and the full conversation history (Stage C). "Prompt engineering" means carefully wording that input to get better outputs. Because the model generates left-to-right with no backspace, the starting conditions steer the entire generation path from the first token.
PROMPT = system instructions + conversation history + your message
       = everything in the context window

Tips: be specific · give context · state the format you want up front
Does Claude read my other conversations or personal data?
+
No. Each conversation is fully isolated. The model only sees the current context window. Past conversations are not included unless you paste them in yourself.
Conversation A (yesterday):  "My name is Priya."

Conversation B (today):  "What is my name?"
Claude: "I don't have that information — this is a fresh conversation."

Exception: if a memory tool is explicitly enabled, summarised context
may be injected into the system prompt. Claude cannot access other
users' conversations.
Why does the model sometimes give wrong answers confidently?
+
The model is a prediction engine, not a search engine. It generates the most statistically likely continuation based on training patterns — it does not verify facts against a database.
WHAT IT DOES:    "What text most plausibly follows this?"
WHAT IT DOESN'T: "Let me verify this against a database."

Obscure, recent, or niche facts → may be wrong or invented.
This is called hallucination.

Claude's Constitutional AI training reduces frequency —
but no model is immune entirely.
What is the difference between a parameter and a token?
+
TOKEN      A piece of text — input or output.
           "What is the capital of France?" = 7 tokens.
           Changes with every message. Runtime data.

PARAMETER  A number in the model's weights — learned during training.
           A 70B model has 70,000,000,000 of them.
           Fixed at inference. Does not change when you chat.

Parameters = synaptic connections (fixed after learning)
Tokens      = thoughts passing through (changing every message)
Why does the same question sometimes get different answers?
+
Temperature and top-p sampling introduce intentional randomness. The model samples from a distribution rather than always picking the top token — so two runs may follow different paths through probability space.
Run 1: "The capital of France is Paris."
Run 2: "Paris is the capital of France."
Run 3: "France's capital is Paris."

All correct. Variation from decoding strategy, not factual inconsistency.
Set temperature = 0 → greedy decoding → fully deterministic.
What happens with very long questions?
+
Everything — your message, system prompt, and conversation history — must fit inside the context window. Exceeding the limit causes truncation or an error.
CONTEXT WINDOW LIMITS (approximate):
  GPT-4:         128,000 tokens  ≈   96,000 words
  Claude:        200,000 tokens  ≈  150,000 words  (≈ a full novel)
  LLaMA 3 (8B):   8,000 tokens  ≈    6,000 words

1 token ≈ 0.75 words in English (Anthropic / OpenAI rule of thumb)
Does the model understand language the way humans do?
+
A genuinely open question in AI research.
DEMONSTRABLY DOES:
  Encodes semantic relationships (embeddings)
  Resolves pronoun references in context
  Applies grammatical patterns consistently
  Recalls factual associations from training
  Reasons through multi-step problems

UNCERTAIN:
  Whether this constitutes "understanding" in a cognitive sense
  Whether it has beliefs, intentions, or awareness

DEFINITIVELY DOES NOT HAVE:
  Persistent memory across conversations (unless given tools)
  → Each new chat is a fresh instance. It does not "forget" —
    there was no continuous "it" between sessions.
  Real-time knowledge (unless given search tools)
  A body, sensory experience, or continuous existence
