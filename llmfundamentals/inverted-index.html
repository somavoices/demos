What is an Inverted Index? concept
A normal forward index maps document → words. An inverted index flips this: word → list of documents containing it. This makes keyword lookup O(1) instead of scanning every document.

Forward Index

Doc 1: ["risk", "management", "bank", "loan"]
Doc 2: ["bank", "river", "flood", "risk"]
Doc 3: ["loan", "approval", "credit", "rate"]

Inverted Index

"risk" →
Doc 1 Doc 2
"bank" →
Doc 1 Doc 2
"loan" →
Doc 1 Doc 3
"management" →
Doc 1
"river" →
Doc 2
The Vocabulary Problem why embeddings win

Query: "financial institution"

Inverted Index

search("financial institution")
No documents found
Neither "financial" nor "institution" appears verbatim in any document — even though Doc A is clearly about banking and finance.

Embedding Search

embed("financial institution")
Doc A (similarity: 0.89)
"bank", "loan", "credit" are semantically close to "financial institution" in vector space — the meaning is captured, not just the spelling.
Limitations at a Glance comparison
Problem Inverted Index Embeddings
Exact keyword ✓ Fast ~ Approximate
Synonyms ("car" vs "automobile") ✗ Misses ✓ Catches
Semantic meaning ✗ None ✓ Captures
Multilingual ✗ Separate indexes ✓ Shared space
Explainability ✓ Transparent ✗ Black box
Speed at scale ✓ Very fast ~ Requires ANN index
Modern Search = Both hybrid
Modern search engines (Elasticsearch, Azure AI Search) use HYBRID search: BM25 (inverted index) for exact match recall + vector embeddings for semantic recall, then fuse the scores with Reciprocal Rank Fusion (RRF). Neither alone is sufficient.
