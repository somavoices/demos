Phase One

Setup and context assembly

Before Claude touches a single file, it chooses a model, reads your project instructions, and loads everything it needs to work effectively.

A
Where Claude Code runs
Terminal · IDE · Browser · Slack
+

Claude Code is available in six environments. The same agentic loop runs everywhere — the difference is only the interface around it.

ENVIRONMENTS:

  $ claude                 ← Terminal CLI  (full-featured, recommended)
  VS Code extension        ← Inline diffs, @-mentions, plan review
  JetBrains plugin         ← IntelliJ, PyCharm, GoLand, WebStorm
  claude.ai/code           ← Browser — no install needed
  Desktop app              ← macOS / Windows / Linux
  Slack integration        ← Tag @Claude Code in threads and PRs

INSTALL (one line):
  macOS/Linux/WSL:   curl -fsSL https://claude.ai/install.sh | bash
  Windows PS:        irm https://claude.ai/install.ps1 | iex

START:
  cd your-project
  claude

Requires Claude subscription (Pro $20/mo · Max $100/mo · Max 20x $200/mo)
or Anthropic Console API key. Terminal CLI also supports third-party providers.
B
Context assembly
CLAUDE.md · Auto-memory · Your prompt
+

Before Claude processes your task, it assembles a context window — everything it will read to understand your project and instructions.

┌──────────────────────────────────────────────────────┐
│  CONTEXT WINDOW                                      │
│                                                      │
│  [System instructions]                              │
│  Built-in Claude Code behaviour and safety rules     │
│                                                      │
│  [CLAUDE.md]                                         │
│  Your project coding standards, architecture,        │
│  preferred libraries, commands, off-limits areas     │
│                                                      │
│  [Auto-memory (MEMORY.md)]                           │
│  Learnings saved from past sessions automatically    │
│  (first 200 lines / 25KB loaded each session)        │
│                                                      │
│  [Your prompt]                                       │
│  "Fix the failing authentication tests"              │
└──────────────────────────────────────────────────────┘
📄
CLAUDE.md is your most powerful lever. Add it to your project root. Every session, Claude reads it first. Invest 30 minutes here and it compounds across every task.

CLAUDE.md files can exist at multiple levels — project root, subdirectories, and your user home (~/.claude/CLAUDE.md for cross-project preferences). Subdirectory files apply only when Claude works inside that folder. Useful for monorepos where each package has different conventions.
Example CLAUDE.md:

  # Auth Service

  Build & test:   make test
  Database:       PostgreSQL via SQLAlchemy ORM
  Auth:           JWT tokens, 24h expiry
  Testing:        pytest only — not unittest

  Never commit API keys. Use .env for all credentials.
  Never modify legacy/ without asking first.
C
Model selection
Sonnet · Opus · Switch mid-session
+

Claude Code uses Claude AI models to understand code and reason about what to do. Different models offer different capability/speed tradeoffs.

claude-sonnet-4-6   Default. Fast. Handles most coding tasks well.
                    Best for: everyday features, bug fixes, refactors.

claude-opus-4-6     Stronger reasoning. Use when you need:
                    - Complex architectural decisions
                    - Challenging multi-system refactors
                    - Hard algorithmic problems
                    - 1M token context window (vs 200K on Sonnet)

claude-haiku-4-5    Fastest and cheapest. Use when you need:
                    - High-volume automated tasks (CI pipelines)
                    - Simple, well-scoped edits
                    - Cost-sensitive batch operations

Switch with:  /model claude-opus-4-6
Start with:   claude --model claude-haiku-4-5

When docs say "Claude decides" or "Claude chooses" —
it is the model reasoning, not a rule engine.
02
Phase Two

The agentic loop

Unlike a chat assistant that responds once, Claude Code runs a continuous loop — gather context, take action, verify results — until your task is complete.

D
Gather → Act → Verify → Repeat
The three-phase loop · ReAct pattern
+

Every task moves through three phases that blend together. A question about your codebase may only need gathering. A bug fix cycles through all three many times. Click each step to see what happens.

01
Gather context
Claude reads files, searches the codebase, runs existing tests, reads error output — building an understanding of the current state before touching anything. For "fix the failing auth tests": Claude runs the test suite first, reads the failures, then locates the relevant source files.
02
Take action
Claude edits files, runs commands, creates new files, stages git changes. Actions are shown as diffs before they are applied. Claude applies changes precisely — surgical edits to specific lines, not full rewrites. You can interrupt at any point to redirect.
03
Verify results
Claude runs tests, reads the output, checks for type errors, verifies the goal was met. If tests fail, it reads the new error messages, adjusts its understanding, and loops back to Take Action with what it learned. It chains dozens of steps, course-correcting as it goes.
04
Task complete (or ask you)
When the task is done (or when Claude needs your input), it stops and reports. You can then approve commits, request changes, or give a new task. You are always part of the loop — Claude works autonomously but stays responsive to your direction at any point.
CONCRETE EXAMPLE: "Fix the failing authentication tests"

  Gather:  Run pytest → 3 tests fail in test_auth.py
           Read error: "jwt.exceptions.ExpiredSignatureError"
           Search for JWT config → find auth.py:47

  Act:     Read auth.py and .env.example
           Identify: expiry set to 0 seconds (typo)
           Edit auth.py line 47: expiry = 86400  # 24h

  Verify:  Run pytest → all 3 tests pass
           Run full suite → 0 regressions

  Done:    Show diff, propose commit message, wait for approval
E
The ReAct pattern
Observe · Reason · Act · Observe...
+

Claude Code follows the ReAct pattern — it interleaves reasoning and acting. Each tool call returns information that feeds directly into the next decision.

REACT (Reasoning + Acting):

  Observe  →  What do I know? What did the last step reveal?
  Reason   →  What does this mean? What should I do next?
  Act      →  Use a tool (read, edit, run, search...)
  Observe  →  What did the tool return?
  Reason   →  Is the task done? What next?
  Act      →  ...  repeat

This means Claude Code adapts based on what it discovers.
It does not follow a rigid script — it reasons at each step.

Alternative patterns: graph routing (LangGraph) and tree search (LATS)
exist but Claude Code uses ReAct for simplicity and low latency.
Each step commits to one action — no backtracking.
💡
Give Claude something to verify against. "Fix the bug and make sure all auth tests pass" is better than "fix the bug" — it gives the loop a clear exit criterion.
03
Phase Three

Tools — how Claude takes action

Without tools, Claude can only respond with text. With tools, it can act. Each tool call returns information that feeds the next step in the loop.

F
File operations
Read · Write · Edit · Rename · Create
+
Read
View any file in your project. Claude reads files to understand context before touching anything — it does not guess.
Write / Edit
Creates new files or makes surgical edits — targeted string replacements, not full rewrites. Shown as a diff before applying.
Rename / Move
Reorganises files. Used in refactors when Claude moves code to a better location or renames for consistency.
Delete
Removes files. Always requires explicit confirmation — never deleted silently.
WRITE RESTRICTION:
  Claude Code can only write to the folder where it was started
  and its subdirectories. It cannot write outside your project
  without explicit permission — a hard constraint, not a default.
G
Search and code intelligence
File search · Content regex · Codebase map
+

Search is how Claude navigates a codebase it has never seen before. It does not pre-read everything — it searches strategically based on your task.

SEARCH TOOLS:
  Find files by pattern:   "all *_test.py files"
  Content search (regex):  "every use of authenticate()"
  Codebase map:            understand module connections without reading all files

CODE INTELLIGENCE (VS Code / JetBrains extension only):
  Live type errors after each edit
  Jump to definition
  Find all references
  Available immediately — no extra configuration needed

Claude does not read your entire codebase on every task.
It navigates intelligently: search first, read only what's needed.
H
Bash / shell execution
Tests · Builds · Git · Any command
+

Claude Code runs terminal commands using your existing tools. It does not have its own versions of git, npm, or pytest — it runs the same commands you would run.

make test                ← run your test suite
pytest auth/             ← scoped test run
git log --oneline -20    ← read recent history
gh pr create             ← open a PR via GitHub CLI
npm run build            ← build the project
docker compose up -d     ← start services
pip install requests     ← install a dependency

SAFETY: Claude asks permission before running shell commands by default.
Dangerous commands (rm -rf, sudo, anything irreversible) always
require explicit approval regardless of your permission settings.
I
Web search and documentation
Real-time lookup · Not frozen at training cutoff
+
Claude can search the web and fetch documentation in real time:

  "What does this error mean?"        → searches Stack Overflow
  "Latest React 19 hook syntax"       → fetches React docs
  "How to configure GitHub Actions"   → reads the docs page

For coding questions, Claude's knowledge is not frozen.
It looks things up when current information would help.

Usage: automatic — Claude decides when a lookup helps.
You can also explicitly ask: "check the docs for this API"
J
Subagents — parallel workstreams
Spawn agents · Isolated branches · Lead orchestrator
+

For large tasks, Claude Code spawns subagents — separate instances that work on different parts simultaneously, coordinated by a lead agent.

EXAMPLE: "Refactor the API layer to async/await"

  Lead agent analyses codebase → divides into 4 modules

  Spawns:
    Subagent A  →  auth module
    Subagent B  →  users module
    Subagent C  →  products module
    Subagent D  →  orders module

    (all running simultaneously, in isolated git worktrees)

  Lead agent:
    Monitors progress
    Merges results
    Runs full test suite
    Resolves any conflicts

Each subagent has its own context window — clean slate per task.
Subagents use git worktrees: isolated branches, no interference.
K
MCP — external service connections
Jira · Slack · Drive · Databases · Custom APIs
+

Model Context Protocol (MCP) is an open standard connecting Claude Code to external tools and data beyond your local filesystem.

MCP-CONNECTED SERVICES:

  Google Drive   →  Read design docs, specs, meeting notes
  Jira / Linear  →  Read and update tickets
  Slack          →  Post updates, read threads
  Figma          →  Read design tokens and components
  Databases      →  Query your database directly
  Custom APIs    →  Any service you expose as an MCP server

EXAMPLE: "Implement the feature described in Jira ticket JR-482"

  Claude reads JR-482 from Jira (via MCP)
  Reads design spec from Google Drive (via MCP)
  Implements the feature across multiple files
  Updates ticket status in Jira (via MCP)
  Opens a PR on GitHub

MCP servers are configured in your Claude Code settings
and connect automatically at session start.
04
Phase Four

Safety, sessions, and delivery

You stay in control. Diffs before edits, permission prompts before commands, checkpoints for undo, and context compaction for long sessions.

L
Permissions and safety modes
Ask · Auto-approve · Sandbox · Yolo
+
Default
Ask everything. Claude prompts before every file write and shell command. Maximum visibility. Best for unfamiliar codebases or sensitive projects.
Auto-approve
Whitelist safe commands. Claude runs whitelisted commands (e.g. npm test, read-only git) without prompting. Reduces friction for trusted operations.
Sandbox (/sandbox)
Filesystem and network isolation. Claude works autonomously within defined boundaries. Best for CI/CD pipelines and automated tasks on untrusted code.
--dangerously-skip-permissions
CLI flag, not a named mode. Skips all permission prompts. Use only in controlled, isolated environments (e.g. sandboxed Docker containers for CI). Never on production systems or anywhere credentials are present. Files in .claude/skills/, .claude/agents/, and .claude/commands/ are always writable without prompting regardless of this flag.
M
Checkpoints and undo
Per-turn snapshots · /undo · Git stash
+

Before every batch of changes, Claude Code creates a checkpoint. If you do not like what it did, /undo rolls back to before that turn.

Your project state: [A]
                     │
Claude plans changes
                     │
Checkpoint created: [A]  ← restore point saved
                     │
Changes applied:    [B]
                     │
You don't like [B]:
  /undo  →  back to [A]

Checkpoints are per-turn (each round-trip with Claude).
Implemented via git stash / git reset under the hood.
Navigate multiple turns back if needed.
N
Context compaction
Five-layer pipeline · /compact · /clear
+

Long sessions fill the context window. Claude Code has a five-layer compaction pipeline. At 85%+, precision drops and hallucinations increase — keeping context healthy is one of the most important habits for reliable results.

0% — work freely
0% 70% — /compact 90% — /clear 
COMPACTION LAYERS (applied in sequence, graduated):
  1. Budget reduction          (always active)
  2. Remove redundant outputs  (tool results already seen)
  3. Summarise old turns       (compress conversation history)
  4. Compress files to diffs   (keep changes, drop originals)
  5. Aggressive summarisation  (last resort)

/compact   manual compaction — run at 70%
/clear     fresh context — mandatory at 90%+
/context   see current context usage
O
Git workflow
Diffs · Commits · Branches · Pull requests
+
GIT WORKFLOW EXAMPLE:
  You:    "Implement password reset and open a PR"

  Claude: 1. git checkout -b feature/password-reset
          2. Read existing auth code for context
          3. Implement across 4 files
          4. make test  →  all pass
          5. Show diff — wait for your review
          6. git add -p && git commit -m "Add password reset flow"
          7. gh pr create --title "Add password reset flow" ...

Claude ALWAYS asks approval before committing.
You see the diff and the commit message before anything is staged.

GIT CAPABILITIES:
  Read:    git status / diff / log / blame
  Write:   git add / commit / stash / checkout
  Create:  git checkout -b (new branches)
  PRs:     gh pr create (via GitHub CLI)
Best practice: Always use feature branches. Never have Claude commit directly to main. Review every diff before accepting — treat all Claude output as code review, not finished product.
P
Skills, hooks, and CI/CD
Slash commands · Automation · Pipelines
+

Skills package repeatable workflows as slash commands. Hooks run shell commands automatically before or after Claude Code actions.

SKILLS — stored in .claude/skills/
  /review-pr       runs your PR review checklist
  /deploy-staging  builds, tests, deploys to staging
  /add-migration   scaffolds a new database migration

HOOKS — shell commands triggered by Claude Code actions
  After every file edit:   eslint --fix on changed file
  Before every commit:     run full test suite; block if failing
  After session ends:      prettier on all changed files

HEADLESS / CI MODE — pipe input, chain with other tools
  git diff main --name-only | claude -p "review for security issues"
  tail -200 app.log | claude -p "summarise any errors"
  claude -p "translate new i18n strings into French, raise a PR"

ROUTINES — scheduled tasks (Anthropic-managed or desktop)
  Morning PR review digest
  Weekly dependency audit
  Nightly CI failure analysis
  Doc sync after every merge

Frequently asked questions

Does Claude Code remember previous sessions?
+
Partially. Auto-memory persists — learnings Claude saves automatically (build commands, your preferences, project patterns). Conversations do not. Each new session starts with a fresh context window; Claude does not recall what you discussed last Tuesday. 
PERSISTS:  Auto-memory (MEMORY.md) · CLAUDE.md
DOES NOT:  Conversation history · File contents · Session state

Workaround: /resume continues the most recent session (if not expired)
            --continue flag does the same via CLI
What can Claude Code actually access?
+
CAN ACCESS:
  Files in your project directory and subdirectories
  Any terminal command you could run (with your permission)
  Your git history, branch state, uncommitted changes
  Your CLAUDE.md and auto-memory
  The web (search and fetch documentation)
  MCP-connected services (if configured)

CANNOT ACCESS:
  Files outside your project directory (without explicit permission)
  Other projects or Claude Code sessions
  Other users' data
  Production systems (unless you provide credentials explicitly)
How is Claude Code different from GitHub Copilot or Cursor?
+
GITHUB COPILOT CLI:
  Command generation. Does not read codebase or run multi-step tasks.

CURSOR:
  Full IDE replacement. Excellent inline editing.
  Operates as an editor, not a terminal agent.

CLAUDE CODE:
  Lives in your terminal (and IDE as an extension).
  Reads your whole codebase, runs commands, manages git.
  Excels at multi-file, multi-step autonomous tasks.
  Pipeable, scriptable, runs in CI/CD.

WHEN TO USE WHICH:
  Inline autocomplete while typing     →  Copilot / Cursor
  Editing one file carefully           →  Cursor
  "Build this feature end to end"      →  Claude Code
  "Fix CI, open a PR, update docs"     →  Claude Code
What happens when Claude Code makes a mistake?
+
Mistakes happen. Community benchmarks have found AI-generated code can introduce significantly more logic errors than human-written equivalents — exact figures vary by study, but the direction is consistent. Always review diffs and run your test suite. 
YOUR SAFETY LAYERS:
  Diff review:   see every change before accepting it
  /undo:         roll back to before the last turn
  Permissions:   risky commands always need approval
  Tests:         give Claude a verifiable success criterion
  Git history:   all committed changes are recoverable

BEST PRACTICE:
  "Fix the auth bug and make sure all auth tests pass"
  is safer than
  "Fix the auth bug"
  because it gives Claude a clear exit criterion.
Is Claude Code safe for production codebases?
+
Yes, with the right practices. 
SAFE PRACTICES:
  Always review diffs before accepting
  Run your test suite after Claude's changes
  Use feature branches — never commit directly to main
  Set permission mode to "ask everything" for sensitive code
  Use /sandbox for CI/CD automation
  Never pass production database credentials unless necessary

COMPLIANCE:
  HIPAA · SOC 2 Type I & II · ISO 27001:2022
  ISO/IEC 42001:2023 · FedRAMP High · UK Cyber Essentials

Anthropic does not train models on your code by default.
See the privacy policy for data retention details.
What should I put in CLAUDE.md?
+
HIGH-VALUE CONTENTS:

  Build and test commands:
    "Run tests with: make test"
    "Build with: npm run build"

  Architecture decisions:
    "All API handlers live in src/handlers/"
    "Use SQLAlchemy ORM — not raw SQL"

  Coding standards:
    "Type hints on all Python functions"
    "Max function length: 30 lines"

  Review checklist:
    "Before any commit: lint, tests, no hardcoded secrets"

  Off-limits:
    "Never modify legacy/ without asking first"
    "Never commit API keys or passwords"

Keep under 500 lines. Long files are read but less precisely followed.

01
Agent, not chatbot
Claude Code takes actions — read, edit, run, commit. The agentic loop keeps running until the task is complete, course-correcting at each step.
02
CLAUDE.md is your lever
Time invested in CLAUDE.md compounds across every session. It is the most reliable way to get consistent, convention-respecting results.
03
Context health matters
Run /compact at 70%, /clear at 90%. Bloated context windows produce less reliable results — treat them like a clean desk policy.
04

Claude Code is not infallible. Review every diff. Run your tests. Treat all output as code review, not finished product.
