01
Phase One

Your message travels to the model

Before any AI processing begins, your text makes a physical journey across the internet to a specialised chip built for parallel computation.

A
You press Enter
HTTPS · Load balancer · GPU
+

Your browser packages your text into a secure HTTPS request — the same protocol used for online banking — and sends it to a data centre. A load balancer routes it to an available GPU: a chip originally built for video games, repurposed for AI because it runs millions of small calculations simultaneously.

YOU PRESS ENTER
      │
      ▼
[ Browser ]  — packages text as HTTPS request
      │  travels over the internet
      ▼
[ Data centre ]  — load balancer routes request
      │
      ▼
[ GPU assigned ]  — processing begins
🔌
Think of it this way: Your sentence is a document dropped in a secure courier envelope (HTTPS). It travels to a sorting office (data centre) and lands with the one specialist (GPU) who can process it — but only after every word is translated from English into the model's native language: pure numbers.
B
Text normalization
Unicode · Whitespace · Encoding
+

Before any AI work begins, your text is cleaned mechanically — extra spaces removed, invisible unicode characters stripped, encoding standardised. No intelligence yet.

RAW:   "What  is the  capital of France?​"
               ^^               ^^           ↑
          double space    double space    invisible char

CLEAN: "What is the capital of France?"

Why it matters: "café" typed two different ways (with a composed vs decomposed character) looks identical but would be treated as different inputs without this step.

C
Context window assembly
System prompt · History · Your message
+

The model never sees just your question. Before tokenization, everything is assembled into one document called the context window.

┌────────────────────────────────────────────────┐
│  CONTEXT WINDOW  (e.g. 200,000 tokens for Claude)  │
│                                                │
│  [System prompt]                              │
│  "You are Claude, a helpful assistant..."      │
│                                                │
│  [Conversation history]                       │
│  User: "Hi"   Assistant: "Hello!"              │
│                                                │
│  [Your current message]                       │
│  "What is the capital of France?"              │
└────────────────────────────────────────────────┘
         │
         ▼  All tokenized together as one input
📋
Think of it this way: The model has no memory between conversations. Every single time it responds, it must re-read the entire context window from scratch — like a chess player who can only see the current board position, not remember the previous game.
02
Phase Two

Text is converted into numbers

The model cannot read English. Everything must become numbers — tokens, then IDs, then meaning-encoded vectors ready for mathematical processing.

D
Tokenization
BPE · Subwords · Misspellings
+

Text is split into tokens — chunks that are usually words but sometimes word-pieces. The tokenizer uses BPE (Byte Pair Encoding), built before model training in a separate step and then permanently frozen.

"What"tok 1
" is"tok 2
" the"tok 3
" capital"tok 4
" of"tok 5
" France"tok 6
"?"tok 7

The space before each word is encoded inside the token — this is how word boundaries are preserved.

Notice " France" became token 6. Follow that token through the next two stages — it becomes the number 3723, then a vector of numbers encoding everything the model knows about France.

Misspellings: A misspelled word may not exist in the vocabulary as a whole — but its pieces usually do. The model reads the parts and infers intent from context.

"beautiful"   → ["beautiful"]             1 token — known word
"beautyful"   → ["beauty"]["ful"]         familiar pieces, same meaning recovered
"definately"  → ["def"]["in"]["ately"]
"recieve"     → ["rec"]["ie"]["ve"]

Exact splits are tokenizer-specific — these are illustrative.
Understanding = (a) familiar pieces  +  (b) surrounding context
🔤
Think of it this way: The model has never learned to read English letters — only token IDs. Tokenization is the translation step at the border crossing: your English sentence gets converted into the model's native vocabulary before it can enter. Without this step, the model would be completely blind to your words.
E
Token IDs
Vocabulary lookup · Numbers only from here
+

Every token is looked up in a fixed vocabulary dictionary of 50,000–128,000 entries and replaced with its unique number. From this point on, the model works only with numbers — never raw text.

VOCABULARY LOOKUP:

  "What"     →  9858
  " is"      →   374
  " the"     →   279
  " capital" →  6864
  " of"      →   315
  " France"  →  3723  ← follow this ID into Stage F
  "?"        →    30

SEQUENCE: [9858, 374, 279, 6864, 315, 3723, 30]

Vocab sizes: GPT-2: 50,257 · GPT-4: 100,256 · Claude: ~100K+ · LLaMA 3: 128,000
📚
Think of it this way: Every word in the model's vocabulary has a permanent seat number — "France" is always seat 3723. The model passes seat numbers around, never the words. Stage F is where each seat number gets its full meaning loaded — like opening the seat's envelope and finding a rich description of everything associated with that word.
F
Embeddings
Meaning vectors · GPS coordinates for words
+

Each token ID is looked up in a learned table — an embedding matrix — and swapped for a vector of hundreds to thousands of numbers. ID 3723 (" France") becomes a specific row in that table. Words with similar meanings end up with similar vectors. The transformer will use these vectors — not the IDs — for all its calculations.

STAGE E output  →  STAGE F: embedding matrix lookup

  ID 9858 ("What")    → [ 0.21, -0.40,  0.88, -0.12,  0.55 ]
  ID  374 (" is")     → [ 0.05,  0.11, -0.22,  0.70, -0.08 ]
  ID 3723 (" France") → [ 0.78,  0.62,  0.15,  0.33, -0.50 ]
                            ↑ this vector carries "France-ness"
  ID 3723 in a different sentence about food?
    → same lookup, same starting vector
    → but attention (Stage I) will reshape it based on context

WHY SIMILAR WORDS GET SIMILAR VECTORS:
  "Paris" ID → [ 0.74,  0.68,  0.18,  0.30, -0.48 ] ← close to France
  "banana" ID → [-0.12, -0.55,  0.71, -0.23,  0.84 ] ← far away

SIZES: GPT-2: 768 · GPT-3: 12,288 · LLaMA 3 (8B): 4,096 · Claude: undisclosed
🗺
Think of it this way: The ID is just an address — it tells the model which row to pick from the embedding table. The vector at that row is what the model actually works with: a fingerprint of meaning, built by learning from billions of sentences. "France" and "Paris" have similar fingerprints. "France" and "banana" do not. Attention in Stage I then updates that fingerprint based on what surrounds it.

⚠ The problem with embeddings: they absorb human bias

Embeddings are learned from real human text — which means they inherit every pattern, stereotype, and bias present in that text. This was first discovered in Word2Vec (an early embedding system from 2013) and remains a live issue in modern LLMs.

WORD2VEC ARITHMETIC — what researchers found:

  "king" − "man" + "woman"  =  "queen"       ← gender relationship encoded
  "Paris" − "France" + "Italy"  =  "Rome"   ← capital relationship encoded

SO FAR SO GOOD. BUT ALSO:

  "programmer" − "man" + "woman"  =  "homemaker"
  "doctor" − "man" + "woman"      =  "nurse"
  "brilliant"  sits close to "man", far from "woman"

WHY THIS HAPPENS:
  Training text reflects historical writing patterns.
  If most articles about programmers used "he," the model
  learns that "programmer" and "man" belong together.
  The model did not invent this bias — it faithfully absorbed it.

This matters because these starting vectors feed into everything the model does. Biased embeddings produce biased translations, biased autocomplete, and biased answers. Modern LLMs use techniques to reduce this — but no model is fully free of it, because no training dataset is.

What you see above is from Word2Vec (2013). Modern LLMs use contextual embeddings — the same word gets different vectors depending on the sentence around it — which reduces but does not eliminate bias inherited from training data.
G
How similarity is measured
Background concept · Cosine · Euclidean
+
This is a background concept — not a step the model takes with your text. Cosine similarity is used in retrieval systems built around LLMs. Inside attention (Stage I), a related but different measure — the raw dot product — is used.
COSINE SIMILARITY  — angle between vectors (retrieval, search)

  "King"   → [ 0.9,  0.3 ]
  "Queen"  → [ 0.8,  0.4 ]   ← similar direction → score: ~0.99
  "Banana" → [-0.2, -0.9 ]   ← opposite direction → score: ~-0.1
  Range: -1 (opposite) to +1 (identical direction)

EUCLIDEAN DISTANCE  — straight-line gap (clustering, retrieval)

  "Paris" → "France" = 1.2  ← close
  "Paris" → "banana" = 8.7  ← far

Inside the model: attention uses raw dot product (Q·K / √d_k)
— related to cosine similarity but NOT identical. See Stage I.
H
Positional encoding
Word order · RoPE in modern models
+

Embeddings carry meaning but no sense of order. "Dog bites man" and "Man bites dog" have identical tokens in different positions — yet mean opposite things. Positional encoding adds a signal to each embedding marking its slot number.

WITHOUT positional encoding:
  ["France", "capital", "is"]  →  order unknown — could mean anything

WITH positional encoding:
  ["France" + pos_1]  [" is" + pos_2]  ["capital" + pos_3]
  "Dog bites man"  ≠  "Man bites dog"  ← now distinguishable

Original transformer (2017): fixed sinusoidal encoding
Modern models (LLaMA, likely Claude): RoPE — learned, relative
Both achieve the same goal: teaching the model about word order.
03
Phase Three

Every word reads every other word

Before transformers, AI models read one word at a time left-to-right and forgot early words by the end of a sentence. The transformer solved this by letting every word look at every other word at once — so "it" can connect back to "trophy" ten words earlier. This block repeats anywhere from 12 times in smaller models to over 100 times in the largest — each pass making the understanding deeper.

I
Attention — words borrowing meaning from each other
How the model figures out what relates to what
+

Every word in your sentence has a simple question to ask of every other word: "how much do you matter to my meaning right now?" The answers are turned into percentages, and each word then borrows meaning from the others in proportion.

This is how the model figures out that "it" means "trophy" and not "suitcase" — even when they are ten words apart. It does not guess; it calculates a score for every possible connection and picks the strongest ones.

How the score is calculated — in plain English

Each word carries its embedding vector from Stage F — its meaning fingerprint. To score how related two words are, the model takes those two fingerprints and measures how well they point in the same direction. Two words whose vectors point similarly score high. Two words whose vectors point in completely different directions score near zero.

SCORING "it" against "trophy" — step by step:

  "it"     has a vector: [ 0.1, -0.3,  0.8,  0.2 ]
  "trophy" has a vector: [ 0.7,  0.5,  0.6, -0.1 ]

  Multiply each pair of numbers together, then add them up:
  (0.1×0.7) + (-0.3×0.5) + (0.8×0.6) + (0.2×-0.1)
  =  0.07  +  -0.15  +  0.48  +  -0.02
  =  0.38  ← a positive score — these words point in a similar direction

  Do the same for "suitcase":
  "suitcase" vector: [ 0.2, -0.8, -0.1,  0.9 ]
  (0.1×0.2) + (-0.3×-0.8) + (0.8×-0.1) + (0.2×0.9)
  =  0.02  +  0.24  +  -0.08  +  0.18
  =  0.36  ← lower — less aligned

  Do this for every word pair. Higher score = more attention paid.
  Then convert all scores to percentages so they sum to 100%.

That's it. No magic — just measuring how much two meaning-fingerprints point in the same direction. The model learned which directions matter for which relationships by reading billions of sentences.

Think of it this way: Before this step, the word "it" in its vector is almost blank — it just means "a pronoun exists here." After attention, it has absorbed 52% of trophy's identity and 31% of "big." It is no longer a placeholder. It is now the specific thing being referred to — and that richer meaning flows into every calculation that follows.
J
The knowledge layer — recalling what the model learned
Where training-time facts get applied to your sentence
+

First — what is a "layer"? Think of it as one full pass of processing. Each transformer block is a layer. Depending on the model's size, there are anywhere from 12 to over 120 of them stacked. Each layer takes the vectors from the previous layer, refines them, and passes them on. By the final layer, each word's vector has been shaped and reshaped until it carries the full meaning of the sentence — not just the word in isolation.

HOW MANY LAYERS DO REAL MODELS HAVE?

  Model                    Layers   What it means
  ─────────────────────────────────────────────────────
  GPT-1 (2018)               12    Early research model
  GPT-2 small (2019)         12    First public GPT
  BERT base (2018)           12    Google's reading model
  BERT large (2018)          24    Larger reading model
  LLaMA 3 8B (2024)          32    Runs on a laptop GPU
  LLaMA 3 70B (2024)         80    High-end server model
  GPT-3 175B (2020)          96    Original ChatGPT base
  LLaMA 3 405B (2024)       126    One of the largest open models
  ─────────────────────────────────────────────────────
  Claude: number of layers not publicly disclosed by Anthropic.

  Each extra layer = one more pass of attention + knowledge retrieval.
  More layers = richer understanding, but slower and costlier to run.

Now — why does Paris fire and not Berlin?

During training, the model read billions of sentences containing "capital of France" followed by or near "Paris." It read almost none saying "capital of France... Berlin." Over time, the weights in this layer adjusted so that the combination "capital + France" produces a strong output signal in the direction of Paris — and a weak one in the direction of Berlin. It was not programmed with this rule. It learned the pattern from repetition.

Think of it like muscle memory built from practice. The model saw "capital of France → Paris" thousands of times. That path became well-worn. "Capital of France → Berlin" was almost never reinforced, so that path stayed faint.

HOW THE KNOWLEDGE LAYER WORKS — with "capital of France":

  BEFORE this layer:
    "capital" vector contains: France context (from attention)
    But it does not yet point toward any city.

  INSIDE the knowledge layer:
    The vector passes through thousands of learned weights.
    Weights trained on "capital + France → Paris" fire strongly.
    Weights trained on "capital + Germany → Berlin" barely fire.
    Weights trained on "capital + UK → London" barely fire.

  AFTER this layer:
    "capital" vector now points toward Paris.
    Not because "Paris" is stored in a box somewhere —
    because the weights collectively push in that direction.

  This is why the model can be wrong on rare facts:
  if "capital of X → Y" barely appeared in training,
  those weights are weak, and the model may guess.
🧠
Think of it this way: Attention told the model what relates to what. This layer tells it what it knows about those things. Without attention first identifying that "capital" is near "France," this layer would not know which stored knowledge to apply. Without this layer, the model could track relationships but could not recall any facts to answer with.
K
Keeping the signal alive across many layers
Why 100+ layers don't destroy what earlier layers learned
+

The transformer runs anywhere from 12 to over 120 layers deep depending on the model. That raises a real problem: if each layer rewrites the vector completely, the original meaning of "France" could be unrecognisable by layer 50. Two mechanisms prevent this.

Residual connection — the "keep a copy" rule

Before each layer does its work, the model takes a copy of the current vector and sets it aside. After the layer finishes, that copy is added back to the result. So no layer can ever fully erase what came before — it can only add a correction on top.

EXAMPLE: the "France" vector passing through one layer

  Vector coming in (after layer 3):
    [ 0.78,  0.62,  0.15,  0.33, -0.50 ]
    "France" — country, western Europe, Paris nearby

  What layer 4's attention adds:
    [ 0.02, -0.05,  0.11,  0.08,  0.03 ]
    (small nudge — "this sentence is a question")

  WITHOUT residual connection:
    Layer 4 overwrites:  [ 0.02, -0.05,  0.11,  0.08,  0.03 ]
    "France-ness" is gone. Layer 5 sees nothing useful.

  WITH residual connection  (input + layer output):
    [ 0.78,  0.62,  0.15,  0.33, -0.50 ]   ← original copy
  + [ 0.02, -0.05,  0.11,  0.08,  0.03 ]   ← layer's correction
  = [ 0.80,  0.57,  0.26,  0.41, -0.47 ]   ← refined, France still present

  Layer 5 receives a richer vector. "France" is still in there.

Layer normalisation — the "steady volume" rule

After many layers of additions, some numbers in the vector can grow very large and some shrink very small. A vector like [412, -0.04, 8831, 1.1] is unstable — the huge value 8831 would dominate all calculations. Layer normalisation rescales every vector after each layer so all values sit in a similar small range, keeping the maths stable throughout the whole stack.

LAYER NORMALISATION — rescaling to a consistent range

  Unstable vector (after many additions):
    [ 412.3,  -0.04,  8831.2,  1.1 ]
    ← 8831 would dominate every calculation that follows

  After layer normalisation (mean=0, range compressed):
    [   0.2,   -0.8,     1.7,  0.0 ]
    ← all values in a similar range — stable input for next layer

  This runs before every sub-layer, every pass, across all layers.
  Without it, the model's maths would become numerically unstable
  by around layer 10 and training would fail entirely.
🔄
Think of it this way: Imagine passing a document through 96 editors. Without the copy rule (residual), each editor rewrites from scratch and the original meaning is gone by editor 10. With the copy rule, each editor only writes their additions in the margin — the original is always there underneath. Layer normalisation is like making sure no editor uses a font so large it covers everything else.
L
Remembering what was already calculated
Why long responses don't slow down token by token
+

The model generates one word at a time. Each new word needs to run attention — comparing itself against every previous word. Without any shortcut, generating word 50 would require recalculating everything for words 1 through 49 all over again. And word 100 would redo everything again. That would be extremely slow.

The solution: save the attention calculations for every word as soon as they are done. When the next word is generated, those saved results are reused — only the newest word needs fresh calculation. This saved store is called the KV cache (K and V are the two sets of numbers each word computes during attention — think of them as "what I offer" and "what I contain").

GENERATING: "The capital of France is Paris."

  Word 1 "The":      calculate, save.  Attend to: [The]
  Word 2 "capital":  calculate, save.  Reuse saved "The".
  Word 3 "of":        calculate, save.  Reuse saved "The", "capital".
  Word 4 "France":    calculate, save.  Reuse all 3 saved.
  Word 5 "is":         calculate, save.  Reuse all 4 saved.
  Word 6 "Paris":      calculate, save.  Reuse all 5 saved.

  Without this:  word 100 recalculates words 1–99 from scratch. Very slow.
  With this:     word 100 only does its own calculation. Fast.

  TRADE-OFF: the saved store takes up GPU memory.
  Very long conversations use more memory — there is a limit.
📝
Think of it this way: Imagine writing an essay and needing to re-read the entire essay from page 1 before writing each new sentence. The KV cache is like keeping sticky notes — every sentence you finish, you write a quick summary note. The next sentence only reads the new notes, not the whole essay again.
04
Phase Four

Turning understanding into a word

After all the transformer layers, each word's vector is rich with meaning. But the model still hasn't produced any output text. Phase 4 is the translation back — from a meaning-vector to an actual word choice. It happens one word at a time, every single step of the response.

M
What the final vector actually holds
The bridge from understanding to output
+

After all the layers have run, the model has one vector for the last position in the sentence. This is not a raw embedding anymore — it has been transformed by every layer of attention and knowledge retrieval. It now encodes something much richer: a compressed summary of the entire question, its context, and what kind of answer is expected.

Think of it as the model's internal answer before any words are chosen. The vector does not say "Paris" — it holds a shape in meaning-space that points strongly toward capital cities, specifically French ones. The steps that follow translate that shape into an actual word.

WHAT THE FINAL VECTOR ENCODES after all layers:

  Input was: "What is the capital of France?"

  Layer 1:   detected it is a question
  Layer 4:   identified "capital" as the key word
  Layer 8:   linked "capital" to "city" concepts
  Layer 15:  connected "France" to European geography
  Layer 24:  associated "capital of France" with a specific city
  Layer 32+: refined confidence, suppressed wrong countries

  FINAL VECTOR:  [ ..., ..., ... ]  (thousands of numbers)
  Meaning:  "the answer is a major European city,
             specifically the capital of France,
             high confidence, proper noun expected"

  This vector has never said "Paris" yet.
  The next stage translates it into word scores.
💡
Think of it this way: After all the layers, the model holds a fully-formed thought — but in its own internal language of numbers. The thought is something like "the answer is a capital city, France, high confidence." Stage N translates that internal thought into a score for every word in the dictionary.
M
Reading the final vector — what word fits here?
From meaning back to vocabulary · One score per word
+

At this point the model has a vector for the last position in the sentence — a list of numbers that encodes everything it understood: the question being asked, the France context, the pattern of "capital city" questions. But a vector of numbers is not a word. The model needs to translate it back.

It does this by asking: "of all the 100,000 words in my vocabulary, which one does this meaning-vector point most toward?" Every word in the vocabulary also has a vector (from Stage F). The model measures how well the current meaning-vector aligns with each vocabulary word's vector — and produces one score per word. These raw scores are called logits.

THE FINAL MEANING VECTOR encodes:
  "question about a capital city, in the context of France"

  Measure alignment with every vocabulary word:

  "Paris"   →  14.8  ← vector strongly aligns — correct answer
  "London"  →   9.2  ← also a capital, but wrong country
  "Berlin"  →   8.1  ← also a capital, but wrong country
  "Rome"    →   7.4  ← also a capital, but wrong country
  "the"     →   3.1  ← not a city
  "banana"  →  -5.3  ← completely unrelated — negative score

  These are raw scores. They cannot be used directly
  because they don't sum to 100% yet. That is Stage N's job.
🔍
Think of it this way: The meaning-vector is like a description — "a capital city in France." The model holds that description up next to every word in its dictionary and asks "how well does this word match the description?" Paris scores highest because it matches best. Banana scores near zero because it doesn't match at all.
N
Converting scores into percentages
Why raw scores aren't enough · Making a fair comparison
+

Stage M produced raw scores — Paris got 14.8, London got 9.2. But raw scores have a problem: you cannot compare them directly or use them to make a choice. Is 14.8 a lot better than 9.2, or just a little? What does 14.8 even mean? There is no scale.

The solution is to convert all scores into percentages that sum to 100%. Now the comparison is clear: Paris has a 72% chance of being the right next word. That is meaningful. This conversion is called softmax.

The important thing softmax does: it amplifies the differences. A score of 14.8 versus 9.2 is not just "a bit higher" — after softmax it becomes six times more likely. The model becomes confident rather than wishy-washy.

Think of it this way: Imagine a panel of judges each holding up a raw score — 14.8, 9.2, 8.1. Those numbers alone don't tell you what to do. But if you convert them to "Paris wins 72% of the vote," you have something you can act on. Softmax is the vote-counting step that turns raw scores into a usable percentage breakdown.
O
How the model picks the actual word
Always pick the top? Or leave room for variety?
+

Stage N produced percentages — Paris 72%, London 12%, Berlin 9%. Now the model must pick one word. The obvious choice is always pick the highest. But that produces robotic, repetitive responses. So there are several strategies, each making a different trade-off between accuracy and variety.

ALWAYS PICK THE WINNER (Greedy)
  Always choose the word with the highest percentage.
  → Always "Paris". Every time. No variation.
  Good for: factual questions, code, structured output.
  Problem: creative writing becomes repetitive and flat.

──────────────────────────────────────────────────────

PICK FROM THE TOP FEW (Top-K)
  Pick randomly from only the top K candidates.
  K = 3: randomly choose from Paris(72%), London(12%), Berlin(9%).
  Could pick London — adds human-feeling variation.
  Problem: on uncertain questions K might include nonsense words.

──────────────────────────────────────────────────────

PICK FROM THE MEANINGFUL CANDIDATES (Top-P)
  Keep adding words until the percentages reach 90% total,
  then pick randomly from just those words.
  Confident answer: Paris(72%) alone hits 72% — very narrow pool.
  Uncertain answer: spreads across many words — wider pool.
  This automatically adjusts to how confident the model is.
  Claude uses this — it is the most natural-feeling approach.

──────────────────────────────────────────────────────

THE CREATIVITY DIAL (Temperature)
  Before any of the above, squash or spread the percentages.

  Low temperature (cautious):   Paris 98%,  London 1.5%  → very safe
  High temperature (creative):  Paris 38%,  London 28%   → more varied

  Claude uses low temperature for code (correctness matters).
  Higher temperature for creative writing (variety matters).
🎲
Think of it this way: Imagine a lottery where Paris has 72 tickets, London has 12, Berlin has 9. Greedy means the highest ticket-holder always wins — predictable but boring. Top-P is like drawing from only the tickets that make sense together. Temperature is like adding or removing tickets before the draw — fewer tickets for Paris means other cities get a fairer shot, making the response feel more creative and varied.
P
Knowing when to stop
How the model decides the answer is complete
+

The model does not have a built-in sense of "I have finished answering." Left alone, it would keep generating words forever — because there is always a next most-likely word. Something has to tell it to stop.

During training, every piece of text the model learned from had an end marker — a special token meaning "this is where this document ends." The model saw this millions of times. It learned that after a complete answer, the next most likely token is that end marker. So when generating, it eventually picks the end marker as the highest-scoring token — and stops.

THE RESPONSE GENERATING:

  "The"  → "capital"  → "of"  → "France"  → "is"  → "Paris"  → "."
       ↓
  Next token scores:
    [END]     →  9.2   ← highest — sentence feels complete
    "It"      →  4.1   ← could keep going
    "The"     →  3.8   ← could start a new sentence

  Model picks [END] → generation stops.

THREE WAYS GENERATION ENDS:
  1. Model picks the end token        ← learned, most natural
  2. A maximum word limit is reached  ← safety cap set by the system
  3. A custom signal is matched       ← e.g. "stop when you see User:"
🔚
Think of it this way: The model learned to stop the same way it learned everything else — from examples. Every book, article, and conversation it trained on had an ending. It absorbed the pattern that a complete, well-formed answer is followed by silence. Generating the end token is the model's way of saying "I'm done here."
Training time — not per-request inference

Before the pipeline runs:
Alignment & RLHF

A raw language model just predicts statistically likely text — not helpful answers. The reason Claude responds directly and honestly is a separate training phase that shaped its weights once, before deployment. RLHF does not run when you press Enter.

RLHF: Human raters compare outputs; a reward model learns human preference; the LLM is fine-tuned to maximise that reward.
Constitutional AI (Claude): A set of principles guides the model's self-critique; AI feedback used alongside human feedback; trained to be helpful, harmless, and honest.
Raw language model
"France is a country in western Europe with a rich history, known for its cuisine, fashion, and the Eiffel Tower..."
Aligned model (Claude)
"The capital of France is Paris."
Same underlying model architecture.
Alignment changed the behaviour, not the structure.

Phase Five

The response travels back to you

One token at a time, streamed live to your browser, then converted back into readable English.

Q
Autoregressive loop
One token at a time · No backspace
+

The model generates one token, appends it to the context, and runs the entire pipeline again. The KV cache means past tokens are not recomputed. This repeats until a stop token is produced.

QUESTION: "What is the capital of France?"

Step 1: → "The"
Step 2: → " capital"
Step 3: → " of"
Step 4: → " France"
Step 5: → " is"
Step 6: → " Paris"
Step 7: → "."
Step 8: → [STOP TOKEN]

OUTPUT: "The capital of France is Paris."

The model cannot revise a word already generated.
It writes left to right — like a typewriter with no backspace.
Prompt wording matters from the very first token.
R
Streaming
Server-Sent Events · Why text appears live
+

Each token is sent to your browser as soon as it is generated via Server-Sent Events (SSE). You start reading while the model is still generating. Total compute time is the same — perceived wait is much shorter.

WITHOUT streaming:  wait several seconds → full answer appears at once

WITH streaming (what actually happens):
  Generate "The"       → send → display "The"
  Generate " capital"  → send → display "The capital"
  Generate " of"        → send → display "The capital of"  ...
  You are reading while the model is still writing.
S
Detokenization
IDs → tokens → readable English

Frequently asked questions

What is a prompt?
+
A prompt is everything inside the context window — your question, the system instructions, and the full conversation history (Stage C). "Prompt engineering" means carefully wording that input to get better outputs. Because the model generates left-to-right with no backspace, the starting conditions steer the entire generation path from the first token.
PROMPT = system instructions + conversation history + your message
       = everything in the context window

Tips: be specific · give context · state the format you want up front
Does Claude read my other conversations or personal data?
+
No. Each conversation is fully isolated. The model only sees the current context window. Past conversations are not included unless you paste them in yourself.
Conversation A (yesterday):  "My name is Priya."

Conversation B (today):  "What is my name?"
Claude: "I don't have that information — this is a fresh conversation."

Exception: if a memory tool is explicitly enabled, summarised context
may be injected into the system prompt. Claude cannot access other
users' conversations.
Why does the model sometimes give wrong answers confidently?
+
The model is a prediction engine, not a search engine. It generates the most statistically likely continuation based on training patterns — it does not verify facts against a database.
WHAT IT DOES:    "What text most plausibly follows this?"
WHAT IT DOESN'T: "Let me verify this against a database."

Obscure, recent, or niche facts → may be wrong or invented.
This is called hallucination.

Claude's Constitutional AI training reduces frequency —
but no model is immune entirely.
What is the difference between a parameter and a token?
+
TOKEN      A piece of text — input or output.
           "What is the capital of France?" = 7 tokens.
           Changes with every message. Runtime data.

PARAMETER  A number in the model's weights — learned during training.
           A 70B model has 70,000,000,000 of them.
           Fixed at inference. Does not change when you chat.

Parameters = synaptic connections (fixed after learning)
Tokens      = thoughts passing through (changing every message)
Why does the same question sometimes get different answers?
+
Temperature and top-p sampling introduce intentional randomness. The model samples from a distribution rather than always picking the top token — so two runs may follow different paths through probability space.
Run 1: "The capital of France is Paris."
Run 2: "Paris is the capital of France."
Run 3: "France's capital is Paris."

All correct. Variation from decoding strategy, not factual inconsistency.
Set temperature = 0 → greedy decoding → fully deterministic.
What happens with very long questions?
+
Everything — your message, system prompt, and conversation history — must fit inside the context window. Exceeding the limit causes truncation or an error.
CONTEXT WINDOW LIMITS (approximate):
  GPT-4:         128,000 tokens  ≈   96,000 words
  Claude:        200,000 tokens  ≈  150,000 words  (≈ a full novel)
  LLaMA 3 (8B):   8,000 tokens  ≈    6,000 words

1 token ≈ 0.75 words in English (Anthropic / OpenAI rule of thumb)
Does the model understand language the way humans do?
+
A genuinely open question in AI research.
DEMONSTRABLY DOES:
  Encodes semantic relationships (embeddings)
  Resolves pronoun references in context
  Applies grammatical patterns consistently
  Recalls factual associations from training
  Reasons through multi-step problems

UNCERTAIN:
  Whether this constitutes "understanding" in a cognitive sense
  Whether it has beliefs, intentions, or awareness

DEFINITIVELY DOES NOT HAVE:
  Persistent memory across conversations (unless given tools)
  → Each new chat is a fresh instance. It does not "forget" —
    there was no continuous "it" between sessions.
  Real-time knowledge (unless given search tools)
  A body, sensory experience, or continuous existence
