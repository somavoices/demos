01
Phase One

Your message travels to the model

Before any AI processing begins, your text makes a physical journey across the internet to a specialised chip built for parallel computation.

A
You press Enter
HTTPS · Load balancer · GPU
+

Your browser packages your text into a secure HTTPS request — the same protocol used for online banking — and sends it to a data centre. A load balancer routes it to an available GPU: a chip originally built for video games, repurposed for AI because it runs millions of small calculations simultaneously.

YOU PRESS ENTER
      │
      ▼
[ Browser ]  — packages text as HTTPS request
      │  travels over the internet
      ▼
[ Data centre ]  — load balancer routes request
      │
      ▼
[ GPU assigned ]  — processing begins
🍽
Analogy: Calling a restaurant order line. Your voice travels through the phone network to a kitchen that starts preparing your order.
B
Text normalization
Unicode · Whitespace · Encoding
+

Before any AI work begins, your text is cleaned mechanically — extra spaces removed, invisible unicode characters stripped, encoding standardised. No intelligence yet.

RAW:   "What  is the  capital of France?​"
               ^^               ^^           ↑
          double space    double space    invisible char

CLEAN: "What is the capital of France?"

Why it matters: "café" typed two different ways (with a composed vs decomposed character) looks identical but would be treated as different inputs without this step.

C
Context window assembly
System prompt · History · Your message
+

The model never sees just your question. Before tokenization, everything is assembled into one document called the context window.

┌────────────────────────────────────────────────┐
│  CONTEXT WINDOW  (e.g. 200,000 tokens for Claude)  │
│                                                │
│  [System prompt]                              │
│  "You are Claude, a helpful assistant..."      │
│                                                │
│  [Conversation history]                       │
│  User: "Hi"   Assistant: "Hello!"              │
│                                                │
│  [Your current message]                       │
│  "What is the capital of France?"              │
└────────────────────────────────────────────────┘
         │
         ▼  All tokenized together as one input
📋
Analogy: Briefing a consultant fresh before every single question — "here are your instructions, here is everything we discussed, here is my new question." They read all of it each time.
02
Phase Two

Text is converted into numbers

The model cannot read English. Everything must become numbers — tokens, then IDs, then meaning-encoded vectors ready for mathematical processing.

D
Tokenization
BPE · Subwords · Misspellings
+

Text is split into tokens — chunks that are usually words but sometimes word-pieces. The tokenizer uses BPE (Byte Pair Encoding), built before model training in a separate step and then permanently frozen.

"What"tok 1
" is"tok 2
" the"tok 3
" capital"tok 4
" of"tok 5
" France"tok 6
"?"tok 7

The space before each word is encoded inside the token — this is how word boundaries are preserved.

Misspellings: A misspelled word may not exist in the vocabulary as a whole — but its pieces usually do. The model reads the parts and infers intent from context.

"beautiful"   → ["beautiful"]             1 token — known word
"beautyful"   → ["beauty"]["ful"]         familiar pieces
"definately"  → ["def"]["in"]["ately"]
"recieve"     → ["rec"]["ie"]["ve"]

Note: exact splits depend on the tokenizer — these are illustrative.
Understanding comes from (a) familiar pieces + (b) sentence context.
🖋
Analogy: Reading "b_autiful" with ink smeared on one letter — context and letter patterns fill the gap. The model does the same with token pieces.
E
Token IDs
Vocabulary lookup · Numbers only from here
+

Every token is looked up in a fixed vocabulary dictionary of 50,000–128,000 entries and replaced with its unique number. From this point on, the model works only with numbers — never raw text.

VOCABULARY LOOKUP:

  "What"     →  9858
  " is"      →   374
  " the"     →   279
  " capital" →  6864
  " of"      →   315
  " France"  →  3723
  "?"        →    30

SEQUENCE: [9858, 374, 279, 6864, 315, 3723, 30]

Vocab sizes: GPT-2: 50,257 · GPT-4: 100,256 · Claude: ~100K+ · LLaMA 3: 128,000
📚
Analogy: A library catalog where every book has a unique number. The model passes catalog numbers around — never the books themselves.
F
Embeddings
Meaning vectors · GPS coordinates for words
+

Each token ID is swapped for an embedding — a list of hundreds to thousands of numbers encoding meaning. Words with similar meanings get similar vectors. These were learned by reading vast text during training.

TOKEN → EMBEDDING (simplified to 5 dims; real: hundreds–thousands)

  " France" → [ 0.78,  0.62,  0.15,  0.33, -0.50 ]
  "Paris"   → [ 0.74,  0.68,  0.18,  0.30, -0.48 ]  ← nearby
  "banana"  → [-0.12, -0.55,  0.71, -0.23,  0.84 ]  ← far away

SIZES: GPT-2: 768 · GPT-3: 12,288 · LLaMA 3 (8B): 4,096 · Claude: undisclosed
🗺
Analogy: Every word as a point on a giant invisible map. "King" and "Queen" sit close together. "Dog" and "Cat" are near each other but far from "King". Embeddings are GPS coordinates on a meaning-map.
G
How similarity is measured
Background concept · Cosine · Euclidean
+
This is a background concept — not a step the model takes with your text. Cosine similarity is used in retrieval systems built around LLMs. Inside attention (Stage I), a related but different measure — the raw dot product — is used.
COSINE SIMILARITY  — angle between vectors (retrieval, search)

  "King"   → [ 0.9,  0.3 ]
  "Queen"  → [ 0.8,  0.4 ]   ← similar direction → score: ~0.99
  "Banana" → [-0.2, -0.9 ]   ← opposite direction → score: ~-0.1
  Range: -1 (opposite) to +1 (identical direction)

EUCLIDEAN DISTANCE  — straight-line gap (clustering, retrieval)

  "Paris" → "France" = 1.2  ← close
  "Paris" → "banana" = 8.7  ← far

Inside the model: attention uses raw dot product (Q·K / √d_k)
— related to cosine similarity but NOT identical. See Stage I.
H
Positional encoding
Word order · RoPE in modern models
+

Embeddings carry meaning but no sense of order. "Dog bites man" and "Man bites dog" have identical tokens in different positions — yet mean opposite things. Positional encoding adds a signal to each embedding marking its slot number.

WITHOUT positional encoding:
  ["France", "capital", "is"]  →  order unknown

WITH positional encoding:
  ["France" + pos_1]  [" is" + pos_2]  ["capital" + pos_3]
  "Dog bites man"  ≠  "Man bites dog"  ← now distinguishable

Original transformer (2017): fixed sinusoidal encoding
Modern models (LLaMA, likely Claude): RoPE — learned, relative
Both achieve the same goal: teaching the model about word order.
03
Phase Three

The transformer reads and understands

The engine of the model. A transformer block runs 12 to 96+ times — each pass deepening the model's grasp of meaning, context, and relationships between words.

I
Multi-head self-attention
Q · K · V · Context mixing · The breakthrough
+

Attention is what makes transformers powerful. Every token simultaneously asks: "which other tokens in this sentence matter to me — and how much?" Each token creates three vectors from itself:

Query (Q)  "What am I looking for?"
Key   (K)  "What do I offer to others looking at me?"
Value (V)  "What is my actual content to contribute?"

Interactive: Click any word to see which others it attends to.

The
trophy
didn't
fit
because
it
was
too
big
"it" strongly attends to "trophy" (52%) and "big" (31%) — the pronoun is resolved.
STEP 1: "it" creates Q: "I am a pronoun — who do I refer to?"
STEP 2: every word creates K and V vectors
STEP 3: score = Q · K / √d_k  (raw dot product — NOT cosine similarity)

  "it" → "trophy"   8.4   →  52%
  "it" → "big"      6.2   →  31%
  "it" → "suitcase" 2.1   →  12%
  "it" → "The"      0.3   →   5%

STEP 4: "it" = 52% of trophy's V + 31% of big's V + ...
Result: "it" now carries the understanding that it means "trophy".

MULTI-HEAD: 12–96 heads run in parallel (model-specific, not universal)
Each head learns different patterns: grammar, facts, coreference...
🏛
Analogy: In a meeting, someone says "the project." You scan the room — who has been talking about which project? — and weight each person's input. Attention does this for every word, across all other words, simultaneously.
J
Feed-forward network
Fact memory · Expand · Activate · Compress
+

After attention mixes context, each token passes through a small neural network independently. If attention is "reading the room," the FFN is "consulting your own memory." Factual associations are believed to live here — though knowledge is distributed, not stored in labelled slots.

EACH TOKEN independently:

  Input vector  [hundreds–thousands of numbers]
       │
       ▼  EXPAND  ×4 (classic) or ×8/3 (SwiGLU models)
       │  e.g.  4,096  →  16,384
       │
       ▼  ACTIVATE  SwiGLU / GELU  — fires only relevant neurons
       │
       ▼  COMPRESS  back to original size
       │  e.g. 16,384  →   4,096

EXAMPLE: "capital" now carries France context (from attention).
The FFN fires "Paris" associations strongly.
"Berlin" and "London" stay weak — wrong country.
(This is a useful intuition, not a literal mechanism.)
🧠
Analogy: After attention tells you "we are talking about France," the FFN is like flipping through your mental notebook — which facts surface given this exact context?
K
Residual connections + layer norm
Signal preservation · Stability · Pre-LayerNorm
+
RESIDUAL CONNECTION

  final_output = sublayer(input) + input
                                    ↑ original signal always preserved

  Without this: signal degrades over 96 layers → training fails.
  With this: each layer learns only the "correction."

PRE-LAYER NORMALISATION  (applied BEFORE each sub-layer in modern models)

  Rescales values: mean = 0, standard deviation = 1.

  Raw:    [412.3,  -0.04,  8831.2,  1.1]
  Normed: [  0.2,   -0.8,     1.7,  0.0]

  Like adjusting a microphone before each speaker —
  prevents any one layer from drowning out the others.
L
KV cache
Avoid recomputation · Speed · Memory cost
+

As the model generates one token at a time, Keys and Values for all past tokens are already computed. The KV cache stores them — only the newest token needs fresh computation each step.

GENERATING: "Paris is the capital of France."

Token "Paris":    compute K₁,V₁ → cache.  Attend: [K₁,V₁]
Token " is":      compute K₂,V₂ → cache.  Reuse K₁,V₁ from cache.
Token " the":     compute K₃,V₃ → cache.  Reuse K₁,V₁,K₂,V₂. ...

WITHOUT cache:  recompute everything each step  → very slow
WITH cache:     only new token computed each step → fast

COST: KV cache uses GPU memory — long conversations use more.
04
Phase Four

A response is chosen word by word

The transformer output is projected onto the full vocabulary, turned into probabilities, and a decoding strategy picks the next token.

M
Unembedding
lm_head · Vector → logits · One score per word
+

A final linear layer (the lm_head) projects the last token's vector onto the entire vocabulary — one raw score called a logit per possible next word.

Final hidden vector  [thousands of numbers]
    ×  lm_head matrix
    ▼
LOGITS  (raw scores, one per vocabulary entry):

  "Paris"   →  14.8  ← very high
  "London"  →   9.2
  "Berlin"  →   8.1
  "Rome"    →   7.4
  "the"     →   3.1
  "banana"  →  -5.3  ← very low
N
Softmax and probability diffusion
Scores → percentages · Exponential amplification
+

Softmax converts raw logits into a probability distribution summing to 100%. It exponentiates each score — amplifying differences dramatically. A logit gap of 5.6 translates to a probability ratio of roughly 6:1.

"Paris"
72.3%
logit 14.8
"London"
12.1%
logit 9.2
"Berlin"
8.8%
logit 8.1
"Rome"
4.4%
logit 7.4
others
2.4%
—

The logit gap between Paris (14.8) and London (9.2) is only 5.6 — yet softmax turns that into a 6:1 probability ratio. Small score differences become large probability differences.

O
Decoding strategy
Greedy · Top-k · Top-p · Temperature
+
GREEDY       Always pick the highest probability word.
             → Always "Paris". Fast. Deterministic. Can be repetitive.

TOP-K        Sample randomly from only the top K candidates.
             K=3: pick from {Paris 72%, London 12%, Berlin 9%}

TOP-P        Keep smallest set of candidates summing to P (nucleus).
             P=0.90:  Paris(72%) + London(12%) = 84%  ... not yet
                      + Berlin(8%) = 92%  ✓  sample from these 3.
             Self-adjusting: confident → narrow set, uncertain → wide.

TEMPERATURE  The creativity dial.
             Low  (0.1): Paris 98%, London 1.5%  → precise, safe
             High (1.5): Paris 38%, London 28%   → varied, surprising

Claude uses top-p + temperature together.
Code responses: low temp.   Creative writing: higher temp.
P
Stop tokens
End of sequence · Three stopping conditions
+

The model does not stop because it runs out of things to say — it stops because it samples a special stop token (a vocabulary entry meaning "done") or because a limit is reached.

STOP TOKENS (examples vary by model family):
  <|endoftext|>   GPT family
  </s>            LLaMA family
  <|eot_id|>      LLaMA 3

STOPPING CONDITIONS:
  1. Model samples a stop token       ← most common
  2. Maximum token limit reached      ← safety cap
  3. Custom stop sequence matched     ← API feature
Training time — not per-request inference

Before the pipeline runs:
Alignment & RLHF

A raw language model just predicts statistically likely text — not helpful answers. The reason Claude responds directly and honestly is a separate training phase that shaped its weights once, before deployment. RLHF does not run when you press Enter.

RLHF: Human raters compare outputs; a reward model learns human preference; the LLM is fine-tuned to maximise that reward.
Constitutional AI (Claude): A set of principles guides the model's self-critique; AI feedback used alongside human feedback; trained to be helpful, harmless, and honest.
Raw language model
"France is a country in western Europe with a rich history, known for its cuisine, fashion, and the Eiffel Tower..."
Aligned model (Claude)
"The capital of France is Paris."
Same underlying model architecture.
Alignment changed the behaviour, not the structure.
05
Phase Five

The response travels back to you

One token at a time, streamed live to your browser, then converted back into readable English.

Q
Autoregressive loop
One token at a time · No backspace
+

The model generates one token, appends it to the context, and runs the entire pipeline again. The KV cache means past tokens are not recomputed. This repeats until a stop token is produced.

QUESTION: "What is the capital of France?"

Step 1: → "The"
Step 2: → " capital"
Step 3: → " of"
Step 4: → " France"
Step 5: → " is"
Step 6: → " Paris"
Step 7: → "."
Step 8: → [STOP TOKEN]

OUTPUT: "The capital of France is Paris."

The model cannot revise a word already generated.
It writes left to right — like a typewriter with no backspace.
Prompt wording matters from the very first token.
R
Streaming
Server-Sent Events · Why text appears live
+

Each token is sent to your browser as soon as it is generated via Server-Sent Events (SSE). You start reading while the model is still generating. Total compute time is the same — perceived wait is much shorter.

WITHOUT streaming:  wait several seconds → full answer appears at once

WITH streaming (what actually happens):
  Generate "The"       → send → display "The"
  Generate " capital"  → send → display "The capital"
  Generate " of"        → send → display "The capital of"  ...
  You are reading while the model is still writing.
S
Detokenization
IDs → tokens → readable English
+
TOKEN IDS:  [791, 6864, 315, 3723, 374, 13210, 13]
              ↓     ↓    ↓     ↓    ↓      ↓    ↓
TOKENS:    "The" "capital" "of" "France" "is" "Paris" "."

JOIN:  "The capital of France is Paris."

Sent back over HTTPS → rendered in your browser.

Spaces and punctuation were encoded inside token pieces throughout — they appear naturally when joined. No post-processing needed.

Frequently asked questions

What is a prompt?
+
A prompt is everything inside the context window — your question, the system instructions, and the full conversation history (Stage C). "Prompt engineering" means carefully wording that input to get better outputs. Because the model generates left-to-right with no backspace, the starting conditions steer the entire generation path from the first token.
PROMPT = system instructions + conversation history + your message
       = everything in the context window

Tips: be specific · give context · state the format you want up front
Does Claude read my other conversations or personal data?
+
No. Each conversation is fully isolated. The model only sees the current context window. Past conversations are not included unless you paste them in yourself.
Conversation A (yesterday):  "My name is Priya."

Conversation B (today):  "What is my name?"
Claude: "I don't have that information — this is a fresh conversation."

Exception: if a memory tool is explicitly enabled, summarised context
may be injected into the system prompt. Claude cannot access other
users' conversations.
Why does the model sometimes give wrong answers confidently?
+
The model is a prediction engine, not a search engine. It generates the most statistically likely continuation based on training patterns — it does not verify facts against a database.
WHAT IT DOES:    "What text most plausibly follows this?"
WHAT IT DOESN'T: "Let me verify this against a database."

Obscure, recent, or niche facts → may be wrong or invented.
This is called hallucination.

Claude's Constitutional AI training reduces frequency —
but no model is immune entirely.
What is the difference between a parameter and a token?
+
TOKEN      A piece of text — input or output.
           "What is the capital of France?" = 7 tokens.
           Changes with every message. Runtime data.

PARAMETER  A number in the model's weights — learned during training.
           A 70B model has 70,000,000,000 of them.
           Fixed at inference. Does not change when you chat.

Parameters = synaptic connections (fixed after learning)
Tokens      = thoughts passing through (changing every message)
Why does the same question sometimes get different answers?
+
Temperature and top-p sampling introduce intentional randomness. The model samples from a distribution rather than always picking the top token — so two runs may follow different paths through probability space.
Run 1: "The capital of France is Paris."
Run 2: "Paris is the capital of France."
Run 3: "France's capital is Paris."

All correct. Variation from decoding strategy, not factual inconsistency.
Set temperature = 0 → greedy decoding → fully deterministic.
What happens with very long questions?
+
Everything — your message, system prompt, and conversation history — must fit inside the context window. Exceeding the limit causes truncation or an error.
CONTEXT WINDOW LIMITS (approximate):
  GPT-4:         128,000 tokens  ≈   96,000 words
  Claude:        200,000 tokens  ≈  150,000 words  (≈ a full novel)
  LLaMA 3 (8B):   8,000 tokens  ≈    6,000 words

1 token ≈ 0.75 words in English (Anthropic / OpenAI rule of thumb)
Does the model understand language the way humans do?
+
A genuinely open question in AI research.
DEMONSTRABLY DOES:
  Encodes semantic relationships (embeddings)
  Resolves pronoun references in context
  Applies grammatical patterns consistently
  Recalls factual associations from training
  Reasons through multi-step problems

UNCERTAIN:
  Whether this constitutes "understanding" in a cognitive sense
  Whether it has beliefs, intentions, or awareness

DEFINITIVELY DOES NOT HAVE:
  Persistent memory across conversations (unless given tools)
  → Each new chat is a fresh instance. It does not "forget" —
    there was no continuous "it" between sessions.
  Real-time knowledge (unless given search tools)
  A body, sensory experience, or continuous existence

01
Prediction engine, not search engine
The model generates the most plausible continuation of text. It does not retrieve facts from a database.
02
Attention is the breakthrough
Every word looks at every other word simultaneously. This is why the model handles long-range relationships that earlier models forgot.
03
Alignment makes it useful
A raw LM just continues text. RLHF and Constitutional AI turn it into a model that answers your question directly and honestly.
